## 1. 项目概述

**flavor-code** 是一个运行在终端里的 AI 编程助手。它不像传统 IDE 插件那样需要图形界面——你只要在命令行里和它对话，它就能阅读代码、搜索文件、修改项目、执行命令、拆分复杂任务并调度多个子 Agent 并行工作，就像一个坐在你旁边的资深程序员。

从架构上讲，它是一个实现了 "ReAct 循环"（Reasoning + Acting）的本地 Agent 框架：接入大语言模型（OpenAI GPT、Anthropic Claude 或兼容服务），在工作区权限范围内安全操作文件，通过分层上下文压缩突破单次模型窗口限制，并通过任务 DAG 实现结构化多步协作。

本报告面向开发者、架构师以及希望深入理解 Agent 系统的技术人员，从目标定位、架构设计、核心子系统到安全策略逐一展开。

**关键设计原则：**

- **组合根集中装配** — `createProductionRuntime()` 负责所有依赖注入，业务模块之间以窄接口连接

## 4. 启动与配置系统

### 4.1 入口路由

`src/cli.tsx` 基于 Commander 实现三个入口：

| 模式 | 命令 | 说明 |
|------|------|------|
| 交互模式 | `flavor` | 全功能 Ink 终端 UI，支持 slash 命令、审批面板、任务进度 |
| 非交互模式 | `flavor --print "..."` | 单次执行，所有审批默认拒绝（deny policy） |
| 会话恢复 | `flavor --resume [id]` | 从 `.flavor/sessions/` 恢复历史会话继续工作 |

非 TTY 且无 `--print` 时退出码 2 并提示用法。

### 4.2 配置加载优先级

```mermaid
flowchart LR
    A["~/.flavor-code/flavor.json<br/>全局配置<br/>优先级: 最低"] --> B[".env<br/>环境变量<br/>叠加到 process.env"]
    B --> C[".flavor/flavor.json<br/>项目配置<br/>优先级: 高"]
    C --> D["CLI 参数<br/>优先级: 最高"]
    D --> E["mergeConfig<br/>深合并"]
    E --> F["interpolate<br/>${VAR} 插值"]
    F --> G["最终 FlavorConfig"]
```

- `${VAR}` 插值优先从 `.env` 取值，其次 `process.env`
- `redactConfig()` 对 `apiKey`、`authorization`、`token` 字段脱敏为 `[redacted]`
- 会话保存前递归扫描并删除敏感字段（`authorization`、`api_key`、`access_token`、`sk-*`）

#### 4.2.1 为什么配置写入不能直接覆盖原文件

配置文件可能同时被两个 Flavor 进程修改。例如，一个进程禁用 MCP server，另一个进程同时启用另一个 server。如果两个进程都先读取旧文件、各自修改后直接覆盖，后写入的进程会把前一个进程的修改吞掉；进程若在写到一半时退出，还可能留下截断 JSON。

`src/config/protected-file.ts` 把这类写入统一成下面的受保护流程：

```mermaid
flowchart TD
    START[准备更新配置] --> LOCK[以 wx 排他模式创建 .lock]
    LOCK -->|锁已存在| WAIT[检查锁年龄和持有进程]
    WAIT -->|持有进程仍存活| RETRY[等待后重试]
    WAIT -->|锁过期且进程不存在| STALE[移除失效锁]
    STALE --> LOCK
    LOCK -->|获得锁| READ[在锁内重新读取主文件]
    READ -->|主文件有效| UPDATE[基于最新值计算更新]
    READ -->|主文件无效| BACKUP_READ[读取并校验 .bak]
    BACKUP_READ -->|有效| UPDATE
    BACKUP_READ -->|也无效| FAIL[失败关闭，不猜测默认值]
    UPDATE --> BACKUP[把覆盖前的有效内容写入 .bak]
    BACKUP --> TEMP[写同目录临时文件并 fsync]
    TEMP --> RENAME[rename 原子替换主文件]
    RENAME --> UNLOCK[仅在锁 token 匹配时删除锁]
```

锁文件保存 `{ pid, nonce }`：默认等待上限为 2 秒，锁超过 30 秒时还会检查 PID 是否仍存活，避免把一个运行较慢但正常的写入者误判为死锁。解锁前再次比对随机 `nonce`，防止误删后来者创建的新锁。

#### 4.2.2 主文件、备份与认证加密

读取不是“JSON 能解析就算成功”，而是同时验证结构和密文认证标签：

1. 先解码主文件；JSON 损坏、结构非法或 AES-GCM 认证失败都视为主文件无效。
2. 再用完全相同的规则解码 `.bak`。
3. 备份有效则继续启动，并在 `sources` 中记录实际使用的 `.bak` 路径。
4. 主备都无效则抛出错误，避免用空配置掩盖凭据损坏或篡改。

全局 `~/.flavor-code/flavor.json` 采用**字段级加密**：名称以 `apiKey`、`authorization`、`token`、`password`、`secret` 等敏感后缀结尾的字符串会保存为 `flavor:v1:<iv>:<tag>:<ciphertext>`。这样非敏感配置仍可人工阅读。首次读到旧明文敏感字段时会生成 32 字节 `.config.key`，随后在锁内自动迁移为 AES-256-GCM 密文；主文件和 `.bak` 都不会再保留该明文。

OAuth Token Store 复用同一套密钥管理和保护写入机制，但采用**整文件加密**，磁盘内容以 `flavor-file:v1:` 开头。整文件加密可以隐藏 provider 名称、过期时间和 scope 等认证元数据。密钥文件以排他创建方式生成，避免两个首次启动的进程生成不同密钥。

## 5. Agent 核心循环

### 5.1 迭代控制

`AgentLoop.run()` 是一个 `async *` 生成器，输出 `AgentEvent` 事件流，由消费方（UI 或子 Agent 调用者）通过 `for await...of` 消费。

```mermaid
stateDiagram-v2
    [*] --> Running: submit(prompt)
    Running --> Warning: 迭代 ≥ maxIterations × 0.8
    Warning --> Running: 继续
    Warning --> Extended: 达到硬限制 & hasActiveProgress()
    Extended --> Running: 追加 extendBy 轮（最多 3 次）
    Extended --> Error: 扩展耗尽
    Running --> Done: 无 tool call + 纯文本
    Running --> ToolExec: 有 tool call
    ToolExec --> Running: 下一轮迭代
    Running --> Overflow: context_overflow（无产出）
    Overflow --> Running: 反应式压缩 + 重试
    Running --> Error: 取消 / 异常
```

**软限制**：迭代达到 80% 时发出 `warning` 事件，告知用户剩余轮数有限。

**硬限制 + 自动扩展**：到达上限时检查 `hasActiveProgress()`——如果存在 `in_progress` 的主任务或 `running` 的子 Agent，自动追加扩展轮数（默认 20 轮，最多 3 次扩展）。子 Agent 不可扩展。

### 5.2 单轮循环详解

```
prepareForModelCall()
  ├── 自动微压缩（快速清理旧工具输出）
  ├── needsCompaction() 判断
  │     └── 完整压缩（模型摘要）
  └── 返回 messages + system
       ↓
BeforeModelCall hook（可 deny）
       ↓
adapter.stream(model, messages, tools)
  ├── text → yield text 事件
  ├── tool-call → 暂存
  ├── usage → 回写 ContextManager
  └── context_overflow → 反应式压缩 + 重试（最多 1 次）
       ↓
AfterModelCall hook
       ↓
有 tool calls?
  ├── 是 → 逐个执行
  │     ├── PreToolUse hook
  │     ├── PermissionEngine.decide()
  │     ├── tool.execute()
  │     ├── PostToolUse hook
  │     └── 失败时 PostToolUseFailure → AuditLogger
  │     └── appendMany(assistant + tool results)
  └── 否 → append(assistant message) → done
```

关键设计细节：

- **整轮暂存**：工具调用整轮暂存后一次性 `appendMany()` 提交 assistant + tool 消息，保证 provider 对话结构有效
- **反应式压缩**：正常请求在尚未产生文本或工具调用时若返回 `context_overflow`，循环在同一 iteration 内强制压缩并重建请求，最多重试一次。若已有可见输出则不重试，避免拼接
- **取消安全**：通过 `AbortSignal` 支持取消，取消后立即合成错误结果并清理

### 5.3 Loop Engineering 外循环

#### 调研结论与边界

截至 2026-07-15，外部资料对 loop engineering 的共同定义不是“把同一个 prompt 无限重试”，而是设计 Agent 外部的控制系统：明确目标、状态、权限、独立验证、反馈路径、停止规则和记忆。论文 *Stop Hand-Holding Your Coding Agent* 将可复用 loop specification 概括为 trigger、goal、verification、stopping rule 和 memory；[Loop Engineering](https://loopengineering.sh/) 也强调它是围绕 Agent 的控制系统。OpenAI 对 [Codex agent loop](https://openai.com/index/unrolling-the-codex-agent-loop/) 的说明则对应本项目已有的内部“模型—工具—观察”循环；[Harness engineering](https://openai.com/index/harness-engineering/) 展示了在其外层通过审查与反馈继续迭代的实践。

因此本项目明确区分两层：

- `AgentLoop` 是一次 worker 内部的 ReAct 循环，模型无工具调用时即可结束。
- `LoopOrchestrator` 是宿主持有的外循环，worker 的“完成”只表示请求验证；只有宿主验证器通过才产生 `succeeded`。

这种分层也用于控制无限循环风险。[When Agents Do Not Stop](https://arxiv.org/abs/2607.01641) 指出，无有效边界的 Agent 循环会放大成本、上下文增长和重复副作用，因此实现必须同时具备预算门槛、可取消性、无进展熔断和具名终态。

#### 命令与运行流程

用户只需调用：

```text
/loop <goal>
```

`loop` 是保留的内置命令，插件或普通 Skill 不能覆盖。方法论由内置 Loop Skill 注入每个新 cycle；调度、验证、预算、持久化和终态判断属于核心运行时，不能由模型自行批准。

```mermaid
flowchart TD
    START["/loop goal"] --> RESOLVE["推断目标意图、隔离与验证命令"]
    RESOLVE -->|无预置验证器| DISCOVERY["一次 verifier-discovery cycle"]
    DISCOVERY -->|仍无法可靠推断| HUMAN["needs_human"]
    DISCOVERY -->|建立项目原生检查| VERIFY
    RESOLVE -->|已推断验证器| WORKER["创建全新 LocalHarness / ContextManager"]
    WORKER --> VERIFY["宿主执行确定性验证器"]
    VERIFY -->|通过| SUCCESS["succeeded"]
    VERIFY -->|失败| PROGRESS{"连续 3 次相同失败且指纹不变?"}
    PROGRESS -->|是| STUCK["no_progress"]
    PROGRESS -->|否| BUDGET{"达到 cycle 或 token checkpoint?"}
    BUDGET -->|否| WORKER
    BUDGET -->|是| CONFIRM["QuestionBridge 请求用户确认"]
    CONFIRM -->|继续| EXTEND["仅增加一个原始 tranche"]
    EXTEND --> WORKER
    CONFIRM -->|拒绝| EXHAUSTED["budget_exhausted"]
    CONFIRM -->|非交互/不可询问| HUMAN
```

#### 验证与隔离推断

验证器按稳定顺序读取 `package.json` 中的 `test`、`typecheck`、`lint`、`build`、`smoke:install`，也识别 `FLAVOR.md` 中明确列出的受信任 npm 命令。命令由宿主的 Shell 工具执行，退出码、截断后的 stdout/stderr 和摘要写入 cycle evidence。若启动时没有确定性命令，调度器不会在 `0 token` 处退出，而是先运行一次 verifier-discovery cycle：worker 检查项目并建立有意义的项目原生检查，宿主随后重新推断并执行。显然无条件成功的脚本（如 `exit 0`、`echo ok`）不会被信任；discovery 后仍没有可靠命令才进入 `needs_human`，不会用模型自述替代证据。

`isolation: "auto"` 的规则是：只读目标使用当前 workspace；代码修改目标及语义不明确的目标要求干净 Git 仓库，并创建 `.worktrees/loop-<id>` 与 `loop/<id>` 分支。`.worktrees/` 只写入仓库本地 `.git/info/exclude`，不修改共享 `.gitignore`。隔离失败时进入 `needs_human`，不会静默降级到主工作区。

#### 分段预算确认

`loop.maxCycles` 与 `loop.maxTokens` 是授权步长，不是整个 loop 的一次性硬上限。初始 checkpoint 等于配置值；达到门槛后只在当前模型流和宿主验证完成的安全边界询问用户，确认后按原始步长增加：

```text
maxTokens = 5,000,000
5,000,000 → 确认 → 10,000,000 → 确认 → 15,000,000 → ...

maxCycles = 20
20 → 确认 → 40 → 确认 → 60 → ...
```

token 统计包括 worker 的所有模型调用以及上下文压缩模型调用；本地 Shell 验证命令不计 token。用户拒绝产生 `budget_exhausted`；非交互模式无法确认时产生 `needs_human`。预算批准记录维度、旧/新 checkpoint、当时累计用量和时间戳。

#### 状态、持久化与当前限制

每次运行写入 `.flavor/loops/<loop-id>/state.json` 和 `events.jsonl`。snapshot 使用临时文件加原子 rename；journal 追加 cycle、预算和终态事件。具名状态包括 `running`、`succeeded`、`failed`、`blocked`、`cancelled`、`budget_exhausted`、`no_progress`、`needs_human`。

当前版本是前台单 loop：不包含后台调度、跨进程自动恢复、自动 merge/push/deploy、容器沙箱或并发 loop 管理。worktree 与分支会保留供用户检查，不自动合并或删除。`blocked` 已进入持久化协议，首版主要通过权限错误和最终失败证据呈现，尚未对所有工具拒绝原因做精细自动分类。

### 5.4 幻觉检测机制

Agent 是概率系统，可能在缺少证据时编造文件路径、命令结果或 API 行为（幻觉）。
flavor-code 实现了一个轻量的三层幻觉检测体系，在每次主 Agent 任务完成前对结果和执行证据进行自动化审查：

**1. 有界过程证据 + Cheap 模型评分** — `EvidenceLedger` 在工具执行时记录调用 ID、工具名、
脱敏参数摘要、成功/失败、错误码和结果短摘要。连续相同事件会折叠，评分时最多保留 24 个事件和
6,000 字符，优先保留验证、修改、失败以及最近事件，不会把完整工具日志塞给评分器。

任务结束时 Cheap 模型在一次请求中同时查看用户 query、最终输出和压缩证据，分别返回：

- `taskAlignment`：是否覆盖用户要求；
- `evidenceGrounding`：最终声明是否有执行证据支撑；
- `processReliability`：是否仍有未解决错误或前后矛盾。

最终置信度由本地固定权重计算：`40% × taskAlignment + 40% × evidenceGrounding + 20% × processReliability`。
工具选择本身不扣分，`Read` 失败后通过 `Shell` 成功读取属于正常恢复。默认阈值仍为 0.7，
低于阈值只产生告警，不再阻断已通过确定性验证的下游任务。

结构化评分关闭修复重试，最多调用模型一次。`hallucination.evaluationTimeoutMs` 控制硬超时，
默认 2,000 ms、允许 100–30,000 ms。超时、模型不可用或格式错误时均 fail-open，直接跳过 LLM 评分。

**2. 工具调用重试监测** — 追踪工具调用失败后的重试行为。默认最大重试 3 次
（匹配项目模型重试设定），同一工具成功执行后自动清零。超过阈值产生 RetryViolation，
报告工具名称、实际重试次数、最大重试次数和最近错误码。

**3. 参数滑动窗口熔断** — 检测"相同工具 + 相同参数"的反复重试死循环。
Hash 计算方式为 SHA256(toolName + JSON(sorted_params))。
默认窗口大小 20，阈值 15。同一 hash 在窗口内出现超过阈值次数即触发电路熔断
（Circuit Breaker），旧条目被推出窗口后自动解除。

**架构集成**：三条路径共享同一 `HallucinationGuard` 实例
（在 `production.ts` 以 subagent model 作为 cheap model 创建）：

- **交互式对话** — `AgentLoop.run()` 中累积文本并记录工具调用/结果。
  模型结束（无 tool call）时触发 `guard.evaluate()`，失败时 yield warning 事件（不阻断对话）。
- **/loop 命令** — `LoopOrchestrator` 验证通过后、声明 success 前触发检查。
  LLM 低分作为成功结果上的附加告警；只有重复失败和确定性循环等本地规则可以将 loop 标记为 failed。
- **子 Agent** — 不介入（`agent !== "main"` 时跳过）。

**关键类型：**

```typescript

  confidence: ConfidenceResult | null;  // 置信度结果（模型不可用时为 null）
```

## 6. 上下文管理与压缩系统

### 6.1 消息布局

每次模型请求时，`ContextManager` 按以下层次拼装消息：

```
┌──────────────────────────────────────┐
│  System Prompt（身份/规则/指令）       │  ← #pinnedMessages() 固定层
├──────────────────────────────────────┤
│  FLAVOR.md（项目指南）                 │  ← 可选
├──────────────────────────────────────┤
│  TaskSnapshot（当前任务状态）          │  ← 可选
├──────────────────────────────────────┤
│  Compact Continuation（压缩摘要续接）  │  ← 注入为 user 消息
├──────────────────────────────────────┤
│  近期完整消息（未压缩部分）            │  ← provider-valid messages
└──────────────────────────────────────┘
```

固定层（system、FLAVOR、TaskSnapshot）不参与压缩持久化，每次由新的 ContextManager 重新生成。

### 6.2 三级压缩策略

```mermaid
flowchart TD
    REQ[每次模型请求前] --> CHK{needsCompaction?}
    CHK -->|否| GO[直接调用模型]
    CHK -->|是| MICRO[一级：微压缩<br/>microcompactMessages]
    MICRO --> CHK2{压力是否解除?}
    CHK2 -->|是| GO
    CHK2 -->|否| AUTO{连续失败 < 3 次?}
    AUTO -->|是| FULL[二级：完整压缩<br/>compactConversation]
    AUTO -->|否| GO_WARN[跳过压缩<br/>熔断保护]
    FULL --> GO

    OVERFLOW[模型返回 context_overflow] -->|无产出| REACT[三级：反应式压缩<br/>重置熔断计数器]
    REACT --> GO_RETRY[重试同一轮]
```

**一级 — 微压缩（Microcompact）**

- 在完整压缩前快速释放空间
- 识别已完成的可压缩工具调用（Read、Shell、Grep、Glob、Edit、Write、ApplyPatch 等 11 种）
- 保留最近 5 个工具结果（`microcompactKeepRecentToolResults`），其余替换为 `[Old tool result content cleared]`
- 失败可回滚，不破坏对话

**二级 — 完整压缩（Model Summary）**

- 调用当前角色模型（空工具集）生成结构化摘要
- 九段式摘要提示词：用户意图与技术决策、文件与代码区、错误与修复、问题解决、待办项、当前工作、下一步、可选分析草稿
- 保留最近 10K–40K token 或 5 条文本消息，不切断 tool call/result
- 自动压缩连续 3 次失败后熔断（避免重复消耗 API）
- 手动 `/compact` 和反应式压缩绕过熔断
- 通过 `snapshot()` / `restore()` 保存和恢复压缩边界

**三级 — 反应式压缩（Reactive）**

- agent loop 在 `context_overflow` 且无可见输出时触发
- 强制完整压缩并重试同一轮
- 重置自动压缩失败计数器

### 6.3 工具输出截断

单次工具消息按 `toolOutputChars`（默认 30,000 字符）进行头尾截断，中间插入截断标记。截断方式为保留字符级头和尾，避免丢失关键信息的同时控制上下文窗口占用。

### 6.4 压力估算模型

- 模型 usage 的 input tokens 回写 ContextManager
- 下一轮以真实 usage + 新增消息估算增量作为压力基线
- 与完整上下文估算取较大值
- 有效窗口 = `windowTokens - reservedOutputTokens`（默认 180K）
- 自动压缩阈值 = 有效窗口 - `autoCompactBufferTokens`（默认 167K）
- 首次调用和刚追加工具结果时使用字符/4 保守估算兜底

## 7. 任务系统

### 7.0 通俗理解：什么是 DAG？

在深入技术细节之前，先用一个生活中的例子理解 DAG（有向无环图）的核心思想。

**类比：做一顿年夜饭。** 年夜饭有 6 道菜，但你的厨房只有 2 个灶台。你不能一上来就炒菜，得先洗菜、切菜——这些步骤有先后顺序：凉拌黄瓜和红烧肉可以同时准备（它们互不依赖），但红烧肉必须先焯水再炖煮（有先后依赖），而摆盘必须在所有菜都做好之后。如果你画一张"先做什么、后做什么"的图，每条线都带箭头（有向）、且不会绕回原点（无环），这就是一张 DAG。

flavor-code 把这个思想用到了 AI 编程中。当你对 flavor-code 说"重构整个 auth 模块"，它不会把 10 个文件一次性全改了——那样容易出错。相反，它会把大任务拆成若干小任务，自动分析哪些任务必须先做、哪些可以并行，然后像"厨师分灶台"一样把任务分配给多个子 Agent 同时执行。

整个任务系统分为两个层次：

| 层次 | 比喻 | 做什么 |
|------|------|--------|
| **TaskPlan**（主计划） | 年夜饭的菜单 | 主 Agent 向用户承诺"我要依次完成 A→B→C→D"，是一个结构化的任务清单 |
| **Task DAG**（子Agent调度） | 分派给帮厨的活 | 把具体工作拆成互相依赖的小任务，交给多个子 Agent 并行执行 |

以下各节从状态机、图结构、调度算法、并发安全等方面，详细介绍这套机制的实现。

### 7.1 两层状态机

Flavor 有两套独立但互补的任务管理机制：

| 层 | 角色 | 工具 | 状态数 | 特点 |
|----|------|------|--------|------|
| TaskPlan | 主 Agent 对用户的结构化承诺 | TaskPlan、TaskUpdate | 6 种 | DAG 依赖、依赖校验、仅主 Agent 可用 |
| TodoWrite | Agent 自用的轻量进度清单 | TodoWrite | 4 种 | 无依赖、最多 50 项、至多 1 项 in_progress |

### 7.2 TaskPlan 状态机

```mermaid
stateDiagram-v2
    [*] --> pending
    pending --> in_progress: TaskUpdate
    pending --> blocked: TaskUpdate
    pending --> cancelled: TaskUpdate
    pending --> pending: 自环允许

    in_progress --> completed: TaskUpdate（依赖需全部完成）
    in_progress --> failed: TaskUpdate
    in_progress --> blocked: TaskUpdate
    in_progress --> cancelled: TaskUpdate
    in_progress --> in_progress: 自环允许

    completed --> completed: 仅自环（终态）
    failed --> failed: 仅自环（终态）
    blocked --> blocked: 仅自环（终态）
    cancelled --> cancelled: 仅自环（终态）
```

**Schema 校验规则：**

- 无重复 ID
- 依赖项必须指向已知 ID
- 不允许重复依赖
- `completed` 节点的所有依赖必须 `completed`
- 最多一个 `in_progress` 任务
- 无循环依赖

`normalizeAbandonedPlan()` 在会话恢复时将所有 `in_progress` 转为 `cancelled`，防止脏状态残留。

### 7.3 DAG 图结构设计

#### 7.3.1 TaskGraph Schema

DAG 的建模单元是 `TaskNode`，由模型通过 `Task` 工具生成，经 Zod Schema 严格校验后传入调度器：

```typescript
// src/agent/planner.ts
const TaskNodeSchema = z.object({
  id: z.string().trim().min(1),                   // 唯一标识
  description: z.string().trim().min(1),           // 任务描述（传给子 Agent 的 prompt 核心）
  dependencies: z.array(z.string().trim().min(1)), // 上游依赖 ID 列表
  expectedOutputs: z.array(z.string().trim().min(1)), // 预期产出（如文件名、函数签名）
  verification: z.array(z.string().trim().min(1)), // 验证命令（如 "npm test"、"tsc --noEmit"）
}).strict();

const TaskGraphSchema = z.object({
  nodes: z.array(TaskNodeSchema),
}).strict().superRefine((graph, context) => {
  // 1. 无重复 ID
  // 2. 所有依赖项必须指向图中已知节点
  // 3. 无重复依赖声明
  // 4. 无循环依赖（Kahn 算法检测）
});
```

#### 7.3.2 图验证：Kahn 拓扑排序检测环

**通俗理解：** Kahn 算法就像在玩"多米诺骨牌"——先找出所有不需要等别人的任务（入度为 0），做完一个就"推倒"它后面的任务；如果最后所有任务都能被推倒，说明图没有环。如果有些牌永远推不倒（被循环依赖卡住了），就说明有环。

`hasCycle()` 使用经典的 Kahn 算法（BFS 拓扑排序），时间复杂度 O(V+E)：

```typescript
// src/agent/planner.ts - hasCycle()
function hasCycle(nodes: readonly TaskNode[]): boolean {
  // indegree: 每个节点还需等待多少个上游节点完成
  const indegree = new Map(nodes.map((t) => [t.id, t.dependencies.length]));
  // dependents: 邻接表（反向边），用于级联减度
  const dependents = new Map<string, string[]>();

  // 构建邻接表
  for (const task of nodes) {
    for (const dep of task.dependencies) {
      (dependents.get(dep) ?? []).push(task.id);
    }
  }

  // 入度为 0 的节点入队（无依赖，立即就绪）
  const ready = nodes.filter((t) => indegree.get(t.id) === 0).map((t) => t.id);
  let visited = 0;

  for (let cursor = 0; cursor < ready.length; cursor++) {
    const id = ready[cursor]!;
    visited += 1;
    // 当前节点"完成"后，所有后继节点入度 -1
    for (const dependent of dependents.get(id) ?? []) {
      const next = (indegree.get(dependent) ?? 0) - 1;
      indegree.set(dependent, next);
      if (next === 0) ready.push(dependent); // 后继就绪
    }
  }
  // 访问节点数 < 总节点数 → 存在环
  return visited !== nodes.length;
}
```

### 7.4 SubagentScheduler 核心算法

**通俗理解：** SubagentScheduler 就像一个"工地工头"。它手里有一张施工图（DAG），知道哪些活必须先干、哪些可以同时干。它手里有 N 个工人（子 Agent），每次挑出"前置活都干完了"的任务分配出去。哪个工人先干完，就立刻给他派下一个能干的活。如果有工人的活干砸了，依赖他的后续工作全部标记为"阻塞"，不会让后面的工人白等。

`SubagentScheduler`（`src/agent/subagents.ts`）是 DAG 执行的引擎，核心逻辑位于其 `run()` 方法中。

#### 7.4.1 完整调度循环

```mermaid
flowchart TD
    START["run(input: TaskGraph)"]
    START --> PARSE["TaskGraphSchema.parse 校验图"]
    PARSE --> INIT["初始化状态映射<br/>所有节点: pending"]

    INIT --> LOOP{"statesHaveWork?<br/>(存在 pending 或 running)"}

    LOOP -->|是| BLOCK["blockDescendants:<br/>标记失败节点的下游为 blocked"]

    BLOCK --> DISPATCH["遍历 nodes，填充到 running 槽位"]
    DISPATCH --> CHECK{"运行槽位 < maxSubagents<br/>且节点 ready?"}
    CHECK -->|是| LAUNCH["节点 → running<br/>启动 runTask 子任务"]
    CHECK -->|否| WAIT

    LAUNCH --> WAIT{"running.size > 0?"}
    WAIT -->|是| RACE["Promise.race(running.values())<br/>等待任一子 Agent 完成"]
    WAIT -->|否| BLOCK2["再次 blockDescendants<br/>处理最后批次"]

    RACE --> COMMIT["记录结果到 results/states Map"]
    COMMIT -->|触发| ON_RESULT["onResult 回调 → publishTaskState"]
    ON_RESULT --> LOOP

    BLOCK2 -->|仍有工作| ERR["throw: 无效状态<br/>无 runnable 任务"]
    BLOCK2 -->|无工作| DONE["return orderedOutcome"]
```

**关键机制：**

1. **事件驱动调度**：使用 `Promise.race(running.values())` 而不是轮询——任一子 Agent 完成后立即评估新的可调度任务，最大化并行度。

2. **槽位控制**：`running.size >= maxSubagents` 时暂停新任务分发，等待已运行任务完成后再填充空槽。

3. **拓扑就绪条件**：一个 `pending` 节点只有在所有 `dependencies` 都是 `"completed"` 时才被认为是 `ready`（`isReady()` 函数）。

4. **清理与取消**：调度器内部创建 `AbortController`，并与外部 `signal` 通过 `AbortSignal.any()` 组合。`finally` 块中 `controller.abort(new Error("Subagent scheduler stopped"))` 确保所有正在运行的子任务收到取消信号，并且 `Promise.allSettled(started)` 等待所有子任务完成清理。

#### 7.4.2 故障传播：blockDescendants

**通俗理解：** 这就像多米诺骨牌的反向传播——如果任务 A 失败了，所有依赖 A 的任务（B、C、D...）自动被标记为"阻塞"，不再浪费资源去执行它们。而且这个阻塞是"传染"的：如果 B 被阻塞了，依赖 B 的任务也会被阻塞，直到所有受影响的链条都停下来。

```typescript
async #blockDescendants(graph, states, results): Promise<void> {
  let changed = true;
  while (changed) {
    changed = false;
    for (const task of graph.nodes) {
      if (states.get(task.id) !== "pending") continue;
      // 只要存在一个上游处于 failed 或 blocked 状态，当前节点被阻塞
      if (!task.dependencies.some((id) =>
        ["failed", "blocked"].includes(states.get(id) ?? "pending"))
      ) continue;
      states.set(task.id, "blocked");
      results.set(task.id, syntheticResult(task, "blocked", "Blocked by a failed dependency"));
      await this.#options.onResult?.(result);
      changed = true; // 级联传播（新 blocked 的节点可能继续阻塞下游）
    }
  }
}
```

传播是**级联的**：A 失败 → B blocked → C blocked，一次 `while` 循环收敛所有影响。

#### 7.4.3 重试机制：最多 2 次尝试

每个子 Agent 任务最多获得 2 次尝试机会：

- **第一次尝试**：正常执行，子 Agent 调用 `TaskOutput` 工具或以 JSON 返回结果
- **Schema 校验**：结果必须通过 `SubagentResultSchema.parse()`，且 `taskId` 必须匹配
- **第二次尝试**（仅当首次校验失败）：prompt 中追加 `"Your previous response was invalid. Return only one strict JSON object."` 修复提示
- **两次均失败**：合成 `status: "failed"` 的 `SubagentResult`，摘要为 `"Invalid structured result after repair"`

```typescript
async #runTask(task: TaskNode, signal: AbortSignal): Promise<Completion> {
  // SubagentStart Hook（可 deny）
  const start = await this.#options.hooks.emit({
    type: "SubagentStart", payload: { taskId: task.id }
  }, signal);
  if (start.decision === "deny") throw ...;

  for (const attempt of [1, 2] as const) {
    signal.throwIfAborted();
    const raw = await awaitWithSignal(
      this.#options.execute(task, { attempt, signal }), signal
    );
    const parsed = SubagentResultSchema.safeParse(raw);
    if (parsed.success && parsed.data.taskId === task.id) {
      completion = { id: task.id, result: parsed.data };
      break; // 校验通过，退出重试
    }
    failure = parsed.success
      ? new Error(`taskId mismatch`)
      : parsed.error;
  }
  // 始终发送 SubagentStop Hook（成功/失败/取消）
  await this.#options.hooks.emit({
    type: "SubagentStop",
    payload: { taskId: task.id, status, error }
  });
}
```

### 7.5 子 Agent 执行生命周期

#### 7.5.1 runChild：独立 harness 构造

每个子 Agent 通过 `runChild()` 函数创建完全隔离的执行环境（`src/production.ts`）：

```typescript
async function runChild(
  harness: LocalHarness, skills: SkillRegistry,
  task: TaskNode, attempt: 1 | 2, signal: AbortSignal,
): Promise<unknown> {
  return harness.runSubagent(task, async (child, childSignal) => {
    // 1. 匹配相关 Skill
    const skill = await skills.match(task.description);
    const additionalContext = skill
      ? `Matched skill: ${skill.name}\n${await skills.loadBody(skill)}`
      : undefined;

    // 2. 构造子 Agent prompt（任务描述 + 预期产出 + 验证命令）
    const repair = attempt === 2
      ? " Your previous response was invalid. Return only one strict JSON object."
      : "";
    const prompt = [
      `Complete task ${task.id}: ${task.description}`,
      `Expected outputs: ${task.expectedOutputs.join("; ")}`,
      `Verification: ${task.verification.join("; ")}`,
      `For completed work, finish by calling TaskOutput...${repair}`,
    ].join("\n");

    // 3. 启动子 Agent Loop
    for await (const event of child.loop.run({ prompt, signal: childSignal, additionalContext })) {
      if (event.type === "error") throw new Error(event.error.message);
      // 4. 捕获 TaskOutput 工具调用 → 直接返回结构化结果
      if (event.type === "tool-end" && event.name === "TaskOutput" && event.result.ok) {
        const completed = subagentResultFromTaskOutput(task.id, event.result.output);
        if (completed !== undefined) return completed;
      }
    }
    // 5. 兜底：从最终 assistant 消息解析 JSON
    return parseFinalSubagentMessage(child.context.snapshot().messages);
  }, signal);
}
```

#### 7.7.2 会话恢复

恢复会话时（`--resume` 标志或 `/resume` 命令），`recovered.tasks` 包含上次保存的完整 DAG 状态：

恢复后的行为：
- **已完成的节点**：状态保留为 `completed`/`failed`/`blocked`，不会重新执行
- **正在运行的节点**（`running`）：`AgentLoop` 的 `hasActiveProgress()` 检测到这些状态，触发主 Agent 的迭代上限自动扩展（3.2 节）
- **待执行的节点**（`pending`）：依赖已满足的节点在下次 `Task` 调用时继续调度

对于 `TaskPlan`（主 Agent 计划），`normalizeAbandonedPlan()` 在恢复时将遗留的 `in_progress` 任务转为 `cancelled`：

```typescript
export function normalizeAbandonedPlan(plan: TaskPlan): TaskPlan {
  return {
    tasks: plan.tasks.map((task) =>
      task.status === "in_progress"
        ? { ...task, status: "cancelled", result: "Execution was abandoned" }
        : { ...task }
    ),
  };
}
```

#### 7.7.3 TaskSnapshot 在上下文中的角色

`TaskSnapshot`（`src/agent/types.ts`）序列化为 JSON 后注入到每次模型请求的 system prompt 固定层（6.1 节消息布局），使模型清楚了解当前 DAG 的执行进度、每个节点的状态、已完成的子 Agent 结果，从而做出有效的下一步决策。

### 7.8 按步骤执行的保障机制

任务系统的核心命题不仅是"描述做什么"，更是**确保 Agent 不跳过关键步骤**。Flavor 从三个层面构建了多层防护：

#### 7.8.1 状态机硬编码 —— 不可绕过的流转约束

TaskPlan 的状态转换由 `isValidTransition()` 硬编码全部合法路径，而非依赖模型自觉：

```
pending ──→ in_progress / blocked / cancelled
in_progress ──→ completed / failed / blocked / cancelled
completed / failed / blocked / cancelled ──→ (终结态，不可再变)
```

```mermaid
flowchart LR
    subgraph 非法路径["非法路径（运行时抛错）"]
        X1["pending → completed ❌"]
        X2["completed → in_progress ❌"]
        X3["failed → pending ❌"]
    end
    subgraph 合法路径["合法路径"]
        L1["pending → in_progress ✓"]
        L2["in_progress → completed ✓"]
        L3["in_progress → failed ✓"]
    end
```

这意味着模型调用 `TaskUpdate` 时如果请求 `pending → completed`，代码层面直接抛出 `Invalid task transition` 错误，**Agent 无法绕过**。

#### 7.8.2 Schema 级校验 —— 提交即检查

每次 `TaskPlan` 提交（创建或替换计划）时，Zod `superRefine` 执行六项强制校验：

| # | 校验项 | 失败后果 | 源码位置 |
|---|--------|---------|---------|
| 1 | 无重复 task id | 拒绝整个 plan | `task-plan.ts:28-30` |
| 2 | 未知依赖（引用了不存在的 task） | 拒绝整个 plan | `task-plan.ts:41-43` |
| 3 | 依赖自身重复引用 | 拒绝整个 plan | `task-plan.ts:44-46` |
| 4 | 同时最多 1 个 `in_progress` | 拒绝整个 plan | `task-plan.ts:33-35` |
| 5 | `completed` 状态的依赖必须全部 `completed` | 拒绝整个 plan | `task-plan.ts:47-56` |
| 6 | 无循环依赖（拓扑排序检测） | 拒绝整个 plan | `task-plan.ts:59-61` |

`TaskUpdate` 在单任务状态变更时额外校验：

| # | 校验项 | 失败后果 |
|---|--------|---------|
| 7 | taskId 必须存在于当前 plan | 抛出 `Unknown task` 异常 |
| 8 | 状态转换必须合法（调用 `isValidTransition`） | 抛出 `Invalid task transition` 异常 |
| 9 | 目标为 `completed` 时所有依赖必须 `completed` | 抛出 `has an incomplete dependency` 异常 |

#### 7.8.3 循环依赖检测

`hasCycle()` 使用标准拓扑排序算法（O(N+E) 复杂度）：

```mermaid
flowchart TD
    A[计算每个节点的入度] --> B[入度为 0 的节点入队]
    B --> C{队列非空?}
    C -->|是| D[出队一个节点, visited++]
    D --> E[将该节点的所有后继入度减 1]
    E --> F{减后入度为 0?}
    F -->|是| G[入队]
    G --> C
    F -->|否| C
    C -->|否| H{visited == 总节点数?}
    H -->|是| I[✅ 无环]
    H -->|否| J[❌ 有环, 拒绝]
```

此检测同时应用于 TaskPlan（主 Agent 计划）和 TaskGraph（子代理 DAG），从代码层面杜绝循环依赖导致的任务死锁。

#### 7.8.4 单任务焦点 —— 防止注意力分散

TodoWrite 和 TaskPlan 各自独立强制执行**同一时间只有 1 个 `in_progress`**。这不仅是一个 UI 展示约定，更是防发散机制：

- 模型无法通过"同时标记多个任务为进行中"来规避按步骤执行
- `AgentLoop` 的自动扩展逻辑也依赖此属性：`hasActiveProgress()` 检测到 `in_progress` 或 `running` 状态时才允许突破硬限制

#### 7.8.5 中断恢复 —— `normalizeAbandonedPlan`

当 Agent 运行中遭遇取消或异常，尚未手动标记完成的任务不能残留为 `in_progress`。`normalizeAbandonedPlan` 自动将所有 `in_progress` → `cancelled`（附 `result: "Execution was abandoned"`），确保下次恢复时状态一致。

```mermaid
    end
```

核心结论：**Agent 的按步骤执行不是靠 prompt 说教，而是靠状态机硬编码 + Schema 校验 + 拓扑排序 + 闭环验证四层机制共同保证**。模型如果尝试跳过步骤——比如不标记 `in_progress` 就直接 `completed`——代码层会直接拒绝，迫使模型回到正确的执行序列上。

### 7.9 DAG 更新正确性保障

**通俗理解：** 你可以把 DAG 更新想象成"银行转账"——不是谁都能随便改账本。flavor-code 有四道关卡保护 DAG 状态的正确性：进门前先验身份（Zod Schema 校验）、操作时核对权限（状态机约束）、修改记录不留痕（纯函数式更新）、还有独立的第三方抽查（TodoWrite 交叉校验）。任何不合规的操作在"第一道门"就会被拦下。

当 Agent 通过 `TaskUpdate` 修改计划状态、或通过 `Task` 工具调度子 Agent 执行结果回写时，整个 DAG 状态的正确性由四层防御体系保证：

#### 第 1 层：Zod Schema 静态校验

所有输入在进入执行逻辑前必须通过严格的 schema 验证。`TaskPlanSchema.superRefine()`（`src/agent/task-plan.ts:24-67`）在解析阶段就进行完整性检查：

- 无重复 task ID
- 无重复依赖声明
- 所有依赖必须指向已存在的 task ID
- `completed` 的任务的所有依赖也必须是 `completed`
- DAG 不得有环（Kahn 拓扑排序算法检测）
- 至多一个 `in_progress`

这意味着一个不合法的 plan 根本无法被创建或提交，在入口处就被拦截。

#### 第 2 层：运行时状态机约束

`updatePlanTask()`（`task-plan.ts:79-103`）在执行更新前：

- 防御性地重新解析输入和当前 plan（两次 Zod 校验）
- 检查 task ID 是否存在（`index >= 0`）
- 调用 `isValidTransition()` 严格限定状态转换路径
- 标记 `completed` 前再次校验所有依赖已完成

状态转换规则硬编码在 `isValidTransition()` 中：

```
pending      → in_progress / blocked / cancelled
in_progress  → completed / failed / blocked / cancelled
终态         → 不可再转换
```

其中一条关键规则是 `if (from === to) return true`，即同一状态的重复提交总是合法的——这为幂等性提供了基础。

#### 第 3 层：纯函数式更新

`updatePlanTask()` 是纯函数——输入 plan + update，输出新的 plan，不修改输入对象。`commit` 回调（`src/production.ts:287`）只是简单赋值 `taskPlan = next`。整个更新路径不涉及 in-place mutation，不会产生部分更新的中间状态。

#### 第 4 层：TodoWrite 交叉校验

`TodoWrite.execute()`（`src/tools/todo-write.ts:39-42`）独立校验"最多 1 个 in_progress"，作为额外安全带。虽然 TodoWrite 面向的是 Agent 自身的子步骤追踪（与 TaskPlan 不同层），但它提供了独立视角的进度一致性检查。

### 7.10 并发安全与幂等性

#### 并发安全：为何不会出现两个流程同时更新同一节点

**通俗理解：** 很多人担心：多个子 Agent 同时干活，会不会出现"两个人同时改同一个文件"的冲突？答案是：不会。因为 JavaScript 是单线程的，所有状态更新都是"一个接一个排队"执行的，就像银行柜台——虽然大厅里有很多人在等，但同一时刻只有一个人在窗口办理业务。

Agent 系统的执行模型天然避免了并发冲突，原因有三：

**1. 同回合内的工具调用是串行的。** 在 `AgentLoop.run()`（`src/agent/loop.ts`）中，即使模型在一次响应中返回多个 tool call，它们的执行也是 `for` 循环串行的：

```typescript
for (let index = 0; index < toolCalls.length; index += 1) {
    const call = toolCalls[index]!;
    const result = await this.#options.runtime.execute(call, ...);
    // 上一个执行完、结果已返回，才进入下一个
}
```

不可能出现同一时刻两个 `TaskPlan`/`TaskUpdate` 调用同时读写 `taskPlan` 变量的情况。

**2. JavaScript 单线程事件循环。** `commit` 回调只是简单赋值 `taskPlan = next`，此操作在 JS 中是原子的——赋值过程不会被另一个操作打断。

**3. 子 Agent 结果回写是顺序的。** `Task` 工具使用 `SubagentScheduler` 并行运行多个子 Agent，但结果收集通过 `Promise.race` 逐个处理（`src/agent/subagents.ts:142`）：

```typescript
const completion = await Promise.race(running.values());
running.delete(completion.id);
results.set(completion.id, completion.result);
states.set(completion.id, completion.result.status);
await this.#options.onResult?.(completion.result);
```

`onResult` 回调虽然 `await` 了，但 scheduler 的 `while` 循环必须等本次 `onResult` 完成才进入下一轮。所以即使 3 个子 Agent 同时跑完，它们的 `onResult` 也是串行执行的。

此外，`Task` 工具在执行前会合并 graph（`production.ts:305-312`）：

```typescript
// Merge new graph nodes into the accumulated task graph so that
// sub-agent statuses from prior Task calls are preserved instead of
// being overwritten.
const priorIds = new Set((taskGraph?.nodes ?? []).map((node) => node.id));
const mergedNodes = [...(taskGraph?.nodes ?? [])];
for (const node of graph.nodes) {
    const index = mergedNodes.findIndex((existing) => existing.id === node.id);
    if (index >= 0) mergedNodes[index] = node;
    else mergedNodes.push(node);
}
taskGraph = { nodes: mergedNodes };
```

旧的 graph 节点状态被保留，新节点被追加或替换，确保多次 `Task` 调用不会互相覆盖状态。

#### 幂等性设计

**TodoWrite 的幂等性。** TodoWrite 每次调用替换整个列表，是纯 "set" 语义。两次传入相同的 `todos` 数组得到完全一致的结果。不累积、不追加。

**TaskPlan 的幂等性。** 同样是替换语义（工具描述明确说 "Create or replace the main-agent plan"）。两次传入相同的 DAG 得到相同结果，没有状态累积。

**TaskUpdate 的幂等性。** 这是最精致的部分。`isValidTransition()` 的关键规则 `if (from === to) return true` 意味着：

- 第一次 `TaskUpdate({ taskId: "1", status: "in_progress" })` 从 `pending → in_progress`，成功
- 第二次同样的调用，`from === to === "in_progress"`，也成功——但结果是同一个状态，行为上幂等
- 但如果任务已经 `completed`，再调用 `in_progress`，转换非法，直接抛错。这防止了对终态的"回退"式误操作

总结：合法范围内的重复调用是安全的（结果不变），非法调用会被立即拒绝（不会产生错误的副作用）。

**Task 工具（子 Agent DAG）的幂等性。** 由于 graph 执行采用合并策略，同一节点 ID 的定义可以被替换但不会重复执行——`SubagentScheduler` 的 `isReady()` 检查确保只有依赖全满足且状态为 `pending` 的节点才会被调度执行。

### 9.3 运行时权限链

```

#### 9.3.1 `auto`：AI 只处理本地规则无法确定的请求

`auto` 并不是把每次工具调用都发送给模型。权限引擎先执行所有确定性规则：安全读取、工作区内写入和经过 cwd/参数路径校验的例行命令直接放行，路径逃逸和硬禁止命令直接拒绝。只有结果仍为 `ask` 时，`ToolRuntime` 才调用 `src/permissions/classifier.ts`。

分类器使用配置的便宜模型，只接收结构化的最小元数据：Agent 类型、工具名、路径、命令、参数和 cwd；不会接收主模型的自然语言解释或思考内容。发送前会清理 Bearer Token、Cookie、URL 用户名密码、敏感查询参数、密钥命令行参数和 PEM 私钥块，并限制数组数量与单字段长度。

分类器必须在默认 2 秒内返回严格的 `{ decision: "allow" | "ask" | "deny", reason }`。结果处理如下：

- `allow`：本次调用继续执行。
- `deny`：本次调用被拒绝，原因返回给 Agent。
- `ask`：进入正常 Hook/UI 审批。
- 超时、模型不可用、结构化输出失败：同样回退正常审批，不会因为分类器故障而静默放行。

#### 9.3.2 `bubble`：子 Agent 不直接向用户交互

创建子 Agent 时，主会话如果处于 `plan`，子 Agent 同样使用 `plan`；其他情况下子 Agent 使用 `bubble`。子 Agent 能在本地确定安全的读取和例行命令继续执行，无法确定的写入、网络或命令请求则复用主会话的 `approve` callback，在主 UI 中显示审批。

因此“冒泡”只改变**由谁完成审批**，不会放宽硬边界：子 Agent 仍不能访问工作区外路径、不能再次调用 `Task` 派生子 Agent，危险命令也不会因为父会话批准能力存在而绕过确定性拒绝。
PreToolUse hook → (可 deny / 修改 input)
  → PermissionEngine.decide()
    → 若 main agent 为 auto 且结果为 ask: 调用便宜模型分类器
      → 分类器超时、失败或返回 ask: 回退正常审批
    → 若仍为 ask: 触发 PermissionRequest hook
      → main agent 调用 approve callback（UI 弹审批面板）
      → bubble 子 Agent 复用主会话 approve callback，把审批冒泡给用户
      → 若 "always": 同分类后续放行（destructive 除外）
  → tool.execute()
  → PostToolUse hook
  → 失败时: PostToolUseFailure hook → AuditLogger
```

### 9.4 Shell 安全分类

`analyzeCommand()` 对每个命令执行多层分析：

- **包装层检测**：`sh -c`、`cmd /c`、`powershell -Command` 等会递归提取内部命令进行再分析
- **模式匹配**：删除命令（`rm -rf`、`del /f`）、磁盘操作（`mkfs`、`format`、`dd`）、仓库破坏（`git push --force`、`git reset --hard`）、权限修改（`chmod 777`）等
- **例行命令**：测试、构建、版本控制、包管理、格式化等常规开发命令
- **降级规则**：不透明命令（复杂管道、重定向）、多语句串联自动降级

### 9.5 工具参数提示 (Tool Hint)

每个工具定义可选实现 `summarize(input) → string | undefined` 方法，用于生成工具调用的简短参数摘要，不包含 ANSI 转义序列。`ToolRuntime.hint()` 在工具开始前解析 input 并调用 summarize，返回纯文本字符串。

`AgentLoop` 将 hint 附加到 `tool-start` 和 `tool-end` 事件的可选 `hint` 字段中，沿事件管道传递至 UI 层。`transcriptReducer` 将 hint 存储在 `TranscriptBlock.hint` 字段，`StatusLine` 组件以灰暗文字 (`dimColor`) 在括号中渲染：` (hint)`。长 hint 使用 `truncate-end` 截断并附加省略号，不会导致换行。

目前已为所有控制工具和搜索工具实现 summarize：

| 工具 | 摘要示例 |
|------|----------|
| Shell | `npm run build`（命令 + 参数拼接） |
| Glob | `pattern: "**/*.ts" in src` |
| Grep | `pattern: /import/ [typescript] in src` |
| TodoWrite | `3/7 done · Implementing cache layer` |
| AskUserQuestion | `2 questions · 6 options` |
| TaskOutput | 直接复用 `summary` 字段内容 |

hint 生成失败时安全降级——输入 schema 校验失败、summarize 抛异常或返回空字符串均视为无 hint，不影响工具执行流程。

### 9.4 LSP 语义代码分析

传统代码搜索（Grep）只能做**文本匹配**——搜 `handleAuth` 可能返回注释、字符串、甚至不同函数里的同名变量。flavor-code 通过 LSP（Language Server Protocol）实现了**语义级代码理解**。

#### 什么是 LSP

LSP 是微软主导的开放协议。核心理念：一种语言只需写一个 Language Server，所有工具（VS Code、Neovim、flavor-code）都能复用。flavor-code 作为 LSP **客户端**，启动各语言的 Language Server 子进程，通过 JSON-RPC 2.0 over stdio 通信，查询代码的语义信息。

#### 三个 LSP 工具

| 工具 | 作用 | 与 Grep 的对比 |
|------|------|---------------|
| `LspFindRefs` | 找到某个符号的所有引用位置 | Grep 搜到的可能是同名变量、注释、字符串；LSP 只返回同一个符号的精确匹配 |
| `LspHover` | 获取光标处符号的类型定义和文档 | Grep 不支持；LSP 直接返回类型签名和 markdown 文档 |
| `LspDiagnostics` | 读取编译器和 linter 的诊断信息 | Grep 不确定是否有错误；LSP 给出精确的行列号和错误描述 |

#### 架构层次

```
Agent（模型）
  ↓ 调用 LspFindRefs / LspHover / LspDiagnostics
ToolRuntime（统一工具执行）
  ↓ execute()
createLspTools(workspace) → ToolDefinition[]
  ↓ 委托
LspManager（接口：findReferences / hover / diagnostics / dispose）
  ↓ 实现
RealLspManager（会话级管理器）
  ↓ 按需懒启动
LspConnection × N（每种语言一个长连接子进程）
  ↓ JSON-RPC 2.0
Language Server 进程（typescript-language-server / pyright / rust-analyzer / gopls）
```

#### 按需懒启动流程

```mermaid
sequenceDiagram

    participant Tools as LSP 工具
    participant Mgr as RealLspManager
    participant Conn as LspConnection
    participant LS as Language Server

    Agent->>Tools: LspFindRefs("src/foo.ts", 10, 5)
    Tools->>Mgr: findReferences(uri, 10, 5)

    Mgr->>Mgr: 从文件后缀判断语言 → typescript

    alt 该语言的连接已存在
        Mgr->>Conn: findReferences(uri, 10, 5)
    else 首次访问
        Mgr->>Mgr: 检查项目是否有 tsconfig.json
        Mgr->>Conn: spawn("typescript-language-server", ["--stdio"])
        Conn->>LS: Content-Length: ...\r\n\r\n{initialize}
        LS-->>Conn: {capabilities: {...}}
        Note over Mgr,Conn: 连接就绪，缓存到 Map
        Mgr->>Conn: findReferences(uri, 10, 5)
    end

    Conn->>LS: Content-Length: ...\r\n\r\n{textDocument/references}
    LS-->>Conn: Content-Length: ...\r\n\r\n[{uri, range}, ...]
    Conn-->>Mgr: LspLocation[]
    Mgr-->>Tools: [{uri: "file:///...", range: ...}]
    Tools-->>Agent: "/project/src/bar.ts:5:2\n/project/src/baz.ts:12:8"
```

#### JSON-RPC 通信协议

Language Server 是一个独立进程，通过 stdin/stdout 通信。每条消息遵循 LSP 标准帧格式：

```
Content-Length: 156\r\n
Content-Type: application/vscode-jsonrpc; charset=utf-8\r\n
\r\n
{"jsonrpc":"2.0","id":1,"method":"textDocument/references","params":{...}}
```

`LspConnection` 内部维护 `LspBuffer` 做流式帧拆分——TCP 是字节流，消息边界靠 `Content-Length` 头部的数字确定，拆分后按 `id` 匹配请求和响应。支持的请求/通知包括：

| 方法 | 方向 | 说明 |
|------|------|------|
| `initialize` | 请求 | 握手，声明客户端能力（hover、references、diagnostics） |
| `initialized` | 通知 | 握手完成 |
| `textDocument/references` | 请求 | 查找引用 |
| `textDocument/hover` | 请求 | 获取悬停信息 |
| `textDocument/diagnostic` | 请求 | 拉取诊断 |
| `shutdown` + `exit` | 请求+通知 | 优雅关闭 |

#### 语言自动检测

`RealLspManager` 根据**文件后缀**匹配语言，再检查工作区内的**根文件**确认是否需要启动：

| 语言 | 文件后缀 | 根文件（任一存在即启动） | Language Server 命令 |
|------|----------|-------------------------|---------------------|
| TypeScript | `.ts`, `.tsx`, `.mts`, `.cts` | `tsconfig.json` | `typescript-language-server --stdio` |
| Python | `.py`, `.pyi`, `.pyx` | `pyproject.toml`, `pyrightconfig.json` | `pyright-langserver --stdio` |
| Rust | `.rs` | `Cargo.toml` | `rust-analyzer` |
| Go | `.go` | `go.mod`, `go.work` | `gopls` |

连接按语言缓存（`Map<language, LspConnection>`），同一种语言的所有文件共享一个长连接，避免重复启动进程。

#### 权限与安全

LSP 三个工具均归类为 `READ_TOOLS`（只读），与 `Read`、`Grep` 同级。工作区内自动放行，无额外审批开销。Language Server 子进程通过 `spawn` 启动，`windowsHide: true`，dispose 时发送 `shutdown` + `exit` 后 `kill` 清理。

### 10.1 19 个事件全集

| 类别 | 事件 | 触发时机 | 决定支持 |
|------|------|----------|----------|
| **会话** | `SessionStart` | 会话开始 | — |

### 10.2 事件格式与处理

```typescript
// 事件格式
{ version: 1, type: "PreToolUse", payload: { toolName, input, ... } }

// 决定格式
{ decision: "allow" | "deny" | "ask", reason?, updatedInput?, additionalContext? }
```

**处理管道：**

1. 按注册顺序遍历处理器
2. `deny` → 短路返回
3. `ask` → 聚合到汇总决定
4. `updatedInput` → 替换后续处理器的 payload（管道模式）
5. `additionalContext` → 跨处理器累积
6. 超时 / 异常 → 按 `failurePolicy` 策略（error/allow/deny/ask）兜底

**Shell 处理器**：通过子进程 stdin 传递 JSON 事件，从 stdout 解析决定，最大输出 1MB，进程树终止跨平台兼容。

## 15. 失败分类与容错

| 层 | 代表性失败 | 处理策略 |
|----|-----------|----------|
| **启动配置** | JSON/Zod 错误、无 provider | 启动错误或可操作诊断；无密钥可进入 UI |

| **UI/进程** | Ctrl+C、关闭期异常 | AbortSignal、幂等 close/dispose |

## 16. 事故上报与 RCA 系统

### 16.1 通俗理解：从"工具挂了"到"找到根因"

在大型项目中，Agent 每天执行成百上千次工具调用。偶尔某个工具会失败——文件不存在、权限不够、Shell 命令执行异常——这些错误往往被淹没在翻滚的屏幕日志里，等真正出问题再回头排查时，现场早已丢失。

事故上报与 RCA（Root Cause Analysis，根因分析）系统的思路是：**每一次工具失败都是现场证据**。失败发生后，系统自动采集失败时的完整现场信息（哪个工具、什么错误、Git 分支和 commit、工作区是否脏），打包成 AlertManager v4 格式的告警，推送到 langgraph-claw 告警管道。P0 级严重错误（工具代码路径损坏）还会触发 langgraph-claw 内的 Auto-RCA 流程——一个 Agent harness 加载 code-rca skill，自动分析错误根因。

换个比喻：就像飞机上的"黑匣子"——平时我们不希望它被触发，但一旦出问题，它能复现完整的事故现场。关键是整个过程**自动进行**，不需要开发者在终端里手动记日志。

### 16.2 告警分级与上报流程

```mermaid
flowchart LR
    subgraph "flavor-code Agent 循环"
        TOOL[工具执行]
        FAIL[PostToolUseFailure]
    end

    subgraph "IncidentReporter"
        COLLECT[采集失败现场信息]
        CLASSIFY[按错误码分级]
        GIT[收集 Git 上下文]
    end

    subgraph "langgraph-claw"
        ALERT[AlertManager /api/otel/alerts]
        P0_RCA["P0 → auto-RCA<br/>agent harness + code-rca skill"]
        P1_STORE["P1 → 存储 + SSE 广播<br/>人工分析"]
    end

    TOOL -->|失败| FAIL
    FAIL --> COLLECT
    COLLECT --> CLASSIFY
    CLASSIFY --> GIT
    GIT --> ALERT
    ALERT -->|P0 critical| P0_RCA
    ALERT -->|P1 warning| P1_STORE
```

错误码与告警级别映射：

| 错误码 | 级别 | 严重度 | 处理方式 |
|--------|------|--------|----------|
| `tool_error` | P0 | critical | 自动触发 RCA，agent harness + code-rca skill 分析根因 |
| `permission_denied` | P1 | warning | 存储 + SSE 广播，人工分析 |
| `hook_denied` | P1 | warning | 存储 + SSE 广播，人工分析 |
| `unknown_tool` | P1 | warning | 存储 + SSE 广播，人工分析 |
| `user_denied` | P1 | warning | 存储 + SSE 广播，人工分析 |
| `approval_required` | P2 | info | 低优先级记录 |
| `invalid_input` | P2 | info | 低优先级记录 |
| 其他 | P3 | none | 不上报 |

每条告报警告负载包含完整的现场信息：

```json
{
  "receiver": "flavor-code",
  "status": "firing",
  "alerts": [{
    "status": "firing",
    "labels": {
      "alertname": "FlavorToolFailure:Shell",
      "severity": "critical",
      "service_name": "flavor-code",
      "error_code": "tool_error",
      "tool": "Shell"
    },
    "annotations": {
      "summary": "[flavor-code] Shell failed: command not found",
      "description": "Tool: Shell\nAgent: main\nError: [tool_error] command not found\nBranch: feature/login\nCommit: a1b2c3d\nWorking tree is clean\nWorkspace: /home/user/my-project\nInput: {\"command\":\"invalid-cmd\"}"
    },
    "startsAt": "2026-07-20T10:30:00Z"
  }],
  "version": "4"
}
```

上报是 **fire-and-forget** 模式——网络失败不会中断 Agent 循环，仅向 stderr 输出 `[incidents]` 日志。

### 16.3 配置与启用

在 `.flavor/flavor.json` 中添加：

```json
{
  "incidents": {
    "enabled": true,
    "webhookUrl": "http://localhost:8000"
  }
}
```

或通过环境变量：

```bash
FLAVOR_INCIDENT_ENABLED=true
FLAVOR_INCIDENT_WEBHOOK_URL="http://localhost:8000"
```

- 默认 webhook 端点：`{webhookUrl}/api/otel/alerts`
- 未启用时 `createIncidentReporter()` 返回零开销的空操作处理器
- 配置优先级：`flavor.json` > 环境变量 > 默认值 `http://localhost:8000`

### 16.4 Git 上下文自动采集

每条告警自动附带工作区的 Git 状态：

```typescript
interface GitContext {
  branch: string | null;   // git rev-parse --abbrev-ref HEAD
  commit: string | null;   // git rev-parse --short HEAD
  dirty: boolean;           // git status --porcelain 非空
}
```

三个命令并行执行（`Promise.all`），超时 3 秒。Git 命令失败时对应字段为 `null`，不阻止告警发送。这份上下文让 RCA 分析时无需手动询问"当时在哪个分支？代码有没有未提交修改？"。

## 17. 安全威胁模型

### 17.1 主要资产

- 工作区代码
- 宿主文件系统
- API 凭据
- 命令执行权
- 对话内容

### 17.2 威胁与缓解

| 威胁 | 缓解措施 |
|------|----------|
| Prompt injection（模型指令注入） | 分层消息设计、固定层不参与压缩 |
| 恶意路径/符号链接 | realpath 校验、逐级包含检查 |
| 模型伪造工具参数 | Zod schema 校验、权限引擎二次决策 |
| Shell 包装层绕过 | 递归命令分析、模式匹配 + 降级规则 |
| 巨大输出/会话 DoS | 输出截断、5MB 会话上限、逐级安全校验 |
| 损坏恢复文件 | 损坏文件隔离、schema 严格校验、不隐式回退 |
| 凭据泄露 | 配置脱敏、会话敏感字段递归删除 |
| 恶意 Skill 资源 | 显式引用校验、stat 快照、路径逃逸检测 |
| 进程内插件 | 信任模型限制（仅加载可信插件） |

### 17.3 剩余风险

- 同进程插件与被批准 shell 具有宿主进程权力
- Provider 会收到模型上下文（包括文件内容）
- 模型可能提出有害但看似常规的操作
- 本地用户仍可读自己的会话文件

## 21. PKCE 认证架构总览

### 21.0 通俗理解：这套认证方案解决了什么问题？

先看一个常见的安全困境：公司买了 LLM API，每个开发者都需要 API Key 才能调用模型。但如果直接把 Key 发给每个人——

- Key 可能被不小心写到代码里、提交到 GitHub
- Key 可能在日志中泄露
- Key 一旦泄漏，攻击者直接用公司的配额疯狂调用，账单暴涨
- 要换 Key，得通知所有人，有人没换就留下安全隐患

**flavor-code 的解决方案是"三权分立"**——把真正的 API Key 锁在服务器机房里，开发者只拿一张"临时通行证"：

> 想象你住酒店。你不会拿着万能房卡到处跑——你去前台登记身份，前台给你一张只能开你房间、3 天后自动失效的房卡。真正的万能卡一直锁在前台保险柜里，从不外借。

在这个类比中：
- **授权服务器** = 酒店前台（验证你的身份、发房卡）
- **JWT Token** = 房卡（3 天有效、只能做授权范围内的事）
- **API 网关** = 楼层服务台（你用房卡进楼层，服务台用万能卡帮你开门）
- **上游 LLM API** = 酒店房间里的设施（真正要用的东西）
- **真实 API Key** = 万能卡（永远不离开服务台）

flavor-code 从 0.1.0 开始内置了 OAuth 2.0 PKCE（Proof Key for Code Exchange）客户端能力，支持与配套的 **flavor-pkce** 项目（授权服务器 + API 网关）协作，实现企业级"无 API Key 暴露"的 LLM 安全访问方案。

#### 长短 Token 双层设计

PKCE 流程中实际使用了两层 Token，一短一长，各司其职：

| 层级 | Token 名称 | 有效期 | 用途 | 类比 |
|------|-----------|--------|------|------|
| **短 Token** | authorization_code | **10 分钟** | 浏览器回调后，flavor-code 用它向授权服务器换取真正的 JWT | 酒店前台给的一张"临时取卡凭条"，只能在前台使用、10 分钟内有效、用一次就作废 |
| **长 Token** | JWT access_token | **3 天** | 签发后 flavor-code 缓存在本地，后续所有 LLM API 请求都用它；3 天内重启无需重新授权 | 酒店房卡，可以反复刷卡开门，3 天后自动失效 |

**为什么需要两个 Token？** 这是安全上的纵深防御。授权码只走浏览器（暴露面大），所以有效期极短（10 分钟）、一次性（用完即标记 `used=1`），即使被截获也几乎没有利用窗口。真正的 JWT 走的是 flavor-code 与授权服务器之间的直连通道（不经过浏览器），安全性更高，有效期可以长达 3 天。

```mermaid
sequenceDiagram
    participant FC as flavor-code
    participant BROWSER as 浏览器
    participant AUTH as 授权服务器
    participant GW as API 网关

    Note over FC,AUTH: ═══ 阶段 1：获取短 Token（authorization_code）═══
    FC->>BROWSER: 打开授权页面（code_challenge 随 URL 传递）
    BROWSER->>AUTH: 用户登录 + 授权
    AUTH->>AUTH: 生成 authorization_code<br/>（10 分钟有效，绑定 code_challenge）
    AUTH-->>BROWSER: 302 重定向，携带 code
    BROWSER-->>FC: 本地回调，传递 code

    Note over FC,AUTH: ═══ 阶段 2：短 Token 换长 Token ═══
    FC->>AUTH: POST /token（code + code_verifier）
    AUTH->>AUTH: 验 code：未使用 ✓、未过期 ✓<br/>验 PKCE：SHA256(code_verifier) == code_challenge ✓
    AUTH->>AUTH: 标记 code 已使用（防重放）
    AUTH-->>FC: JWT access_token（3 天有效）

    Note over FC,GW: ═══ 阶段 3：长 Token 日常使用 ═══
    FC->>GW: API 请求（Authorization: Bearer JWT）
    GW->>GW: 验签 JWT ✓ → 替换为真实 API Key
    GW-->>FC: LLM 流式响应

    Note over FC,AUTH: ═══ 3 天后：长 Token 过期 ═══
    FC->>FC: 检测 expiresAt - 60s < now<br/>→ 自动清除缓存<br/>→ 重新触发阶段 1
```

**当前版本的 Token 刷新策略**：没有使用 OAuth refresh_token 机制。JWT 过期前 60 秒 flavor-code 自动丢弃缓存，下次启动时重新走完整的 PKCE 流程（弹出浏览器 → 用户授权 → 获取新 JWT）。设计如此，因为 PKCE 流程在本地回环完成、用户体验轻量，且避免了 refresh_token 长期存储带来的安全风险。

### 21.1 架构拓扑

```mermaid
flowchart TD
    FC["flavor-code CLI<br/>（TypeScript 客户端）"]
    AUTH["授权服务器<br/>auth-server (Python/FastAPI)<br/>端口 8091"]
    GW["API 网关<br/>api-gateway (Python/FastAPI)<br/>端口 8092"]
    UP["上游 LLM<br/>（如 OpenAI API）"]

    FC -->|1. GET /authorize?code_challenge=...<br/>浏览器回调| AUTH
    AUTH -->|2. 302 redirect → callback<br/>code + state| FC
    FC -->|3. POST /token<br/>code + code_verifier| AUTH
    AUTH -->|4. JWT access_token| FC
    FC -->|5. 所有 API 请求<br/>Authorization: Bearer JWT| GW
    GW -->|6. 校验 JWT → 替换为<br/>真实 API Key| UP
    UP -->|7. SSE/JSON 流式响应| GW
    GW -->|8. 透明转发| FC
```

### 21.2 为什么需要 PKCE + 网关

在团队或企业环境中，直接让每个开发者持有 LLM API Key 存在以下风险：

- **凭据泄露**：API Key 一旦被日志记录、代码提交或恶意窃取，攻击者可直接消费企业配额
- **审计空白**：不知道谁在什么时间使用了多少 token
- **密钥轮换困难**：API Key 泄漏后需要通知所有人更换

**PKCE + API 网关** 方案将真正的 API Key 封存在网关后端：

1. 用户通过浏览器登录授权服务器（用户名/密码）
2. 授权服务器验证身份后签发 **JWT**（JSON Web Token）
3. flavor-code 使用 JWT 向 API 网关发起 LLM 请求
4. 网关验证 JWT 后**替换 Authorization header** 为真实 API Key，转发到上游 LLM
5. 真实 API Key **永不离开网关**，不在网络上传输，不出现在任何客户端

## 22. PKCE 客户端实现（flavor-code）

**通俗理解：** 这一章讲的是"你手里的 flavor-code 是怎么自动弄到那张房卡的"。整个流程对用户来说是透明的——你只需要在命令行的指引下用浏览器登录一次，剩下的 token 获取、缓存、过期刷新全部由 flavor-code 自动完成。整个过程就像"手机 App 用微信登录"：点击登录 → 跳转浏览器 → 确认授权 → 回到 App，只不过这是为命令行设计的。

### 22.1 客户端架构

flavor-code 的 PKCE 客户端由三个模块组成，呈单向依赖关系：

```
src/config/schema.ts ────→ 定义 OAuth 配置字段
src/production.ts ───────→ 启动时调用 OAuth 流程获取 API Key
src/auth/oauth.ts ───────→ OAuthCallbackAuthProvider（PKCE 核心）
src/auth/store.ts ───────→ 文件系统 Token 持久化
src/auth/types.ts ───────→ AuthProvider / AuthResult / ApiKeyAuthProvider
```

### 22.2 配置 Schema 扩展

`ProviderConfigSchema`（`src/config/schema.ts`）在原有 `apiKey` 字段基础上新增了 OAuth PKCE 字段：

```typescript
apiType: z.enum(["openai", "anthropic"]).optional(),  // API 协议类型
authorizationUrl: z.string().url().optional(),         // 授权端点
tokenUrl: z.string().url().optional(),                 // Token 端点
clientId: z.string().min(1).optional(),                // 客户端标识
scope: z.string().optional(),                          // 权限范围
```

### 22.3 Provider 启动决策树

**通俗理解：** 启动时 flavor-code 会按以下顺序"想办法弄到 Key"：手里本来就有 Key 就直接用 → 没有就去后台领一张（PKCE 流程）→ 连后台地址都没配置就试试环境变量 → 都没有才报错。这样设计的意义是：无论你是个人开发（直接写 apiKey）、团队用户（PKCE）、还是 CI 环境（环境变量），都能自动适配。

`registerConfiguredAdapters()`（`src/production.ts`）按以下优先级确定 API Key：

```mermaid
flowchart TD
    START["Provider 配置"] --> A{有 apiKey?}
    A -->|是| USE_KEY["直接使用 apiKey"]
    A -->|否| B{OAuth 字段完整?<br/>authorizationUrl + tokenUrl + clientId}
    B -->|是| PKCE["执行 PKCE 流程<br/>OAuthCallbackAuthProvider"]
    B -->|否| C{有 OAUTH_* 环境变量?}
    C -->|是| PKCE_DEFAULT["使用内建默认值<br/>执行 PKCE 流程"]
    C -->|否| D{有对应 API Key 环境变量?}
    D -->|是| USE_ENV["使用环境变量"]
    D -->|否| ERR["诊断错误：缺少凭据"]
```

**关键逻辑**：`resolveOAuthConfig()` 函数在没有显式 apiKey 和 OAuth 配置时，自动从环境变量 `OAUTH_AUTHORIZATION_URL`、`OAUTH_TOKEN_URL`、`OAUTH_CLIENT_ID`、`OAUTH_SCOPE` 读取内建默认值，无需用户手动填写。

### 22.4 OAuthCallbackAuthProvider 流程

**通俗理解：** 下面是 PKCE 最核心的"握手"过程。它的精妙之处在于：浏览器和授权服务器之间传递的 `code_challenge` 是哈希值，即使被截获也无法还原出 `code_verifier`；而真正用于换 Token 的 `code_verifier` 只在 flavor-code 和授权服务器之间直传，不经过浏览器。这就是 PKCE 安全性的关键——"中间人截到挑战也没法用"。

`OAuthCallbackAuthProvider`（`src/auth/oauth.ts`）实现了完整的 RFC 7636 PKCE 客户端流程：

```mermaid
sequenceDiagram
    participant STORE as Token Store<br/>(auth.json)
    participant PROV as OAuthCallbackAuthProvider
    participant BROWSER as 系统浏览器
    participant SRV as 临时 HTTP Server<br/>(127.0.0.1:随机端口)
    participant AUTH as 授权服务器

    PROV->>STORE: load() 检查缓存 Token
    alt Token 有效（过期时间 - 60s > 当前时间）
        STORE-->>PROV: { accessToken, expiresAt }
        PROV-->>调用方: AuthResult { Authorization: Bearer JWT }
    end

    STORE-->>PROV: 无有效 Token

    PROV->>PROV: 生成 code_verifier (randomBytes(32) → base64url, 43字符)
    PROV->>PROV: 计算 code_challenge (SHA256(code_verifier) → base64url)
    PROV->>PROV: 生成 state (randomBytes(32) → base64url, 防CSRF)
    PROV->>SRV: 启动临时 HTTP Server 监听 127.0.0.1:随机端口/callback

    PROV->>PROV: 构造 authorizationUrl ?response_type=code &client_id=... &redirect_uri=... &code_challenge=... &state=...

    PROV->>BROWSER: openBrowser(authUrl)
    BROWSER->>AUTH: GET /authorize?...（用户登录+授权）
    AUTH->>BROWSER: 302 重定向到 callback
    BROWSER->>SRV: GET /callback?code=...&state=...

    SRV->>SRV: 校验 state 一致（防 CSRF）
    SRV->>SRV: 校验 code 非空
    SRV-->>PROV: 返回 authorization_code

    PROV->>AUTH: POST /token (grant_type, code, code_verifier, redirect_uri, client_id)

    alt PKCE 验证成功
        AUTH-->>PROV: { access_token: eyJ..., token_type: Bearer, expires_in: 259200 }
        PROV->>STORE: save({ accessToken, refreshToken?, expiresAt, scope })
        PROV-->>调用方: AuthResult { Authorization: Bearer eyJ... }
    else 验证失败
        AUTH-->>PROV: { error: invalid_grant }
        PROV-->>调用方: throw Error
    end
```

**安全设计要点：**

1. **code_verifier 熵**：32 字节随机数 → 256 位熵，暴力破解不可行
2. **S256 散列**：SHA256(code_verifier) → code_challenge，即使 challenge 泄露也无法还原 verifier
3. **state 防 CSRF**：每次流程生成新的 32 字节随机 state，回调时必须原样匹配
4. **重定向 URI 锁定**：固定 `http://127.0.0.1:{随机端口}/callback`，仅接受本地回环
5. **5 分钟超时**：防止悬挂的授权流程无限等待（`PKCE_TIMEOUT_MS = 300_000`）
6. **Token 预刷新**：过期前 60 秒自动丢弃缓存，触发新的 PKCE 流程
7. **同时只允许一个流程**：`#pendingFlow` 防并发授权

### 22.5 Token 持久化

`createFileTokenStore()`（`src/auth/store.ts`）将 Token 持久化到 `~/.flavor-code/auth.json`：

下面是**解密后的逻辑结构**，便于说明数据模型：

```json
{
  "openai": {
    "accessToken": "eyJhbGciOiJSUzI1NiIs...",
    "expiresAt": "2026-07-17T12:00:00.000Z",
    "scope": "models:read models:use"
  }
}
```

磁盘上的 `auth.json` 不保存上述明文，而是一个完整的认证加密信封：

```text
"flavor-file:v1:<base64 iv>:<base64 auth tag>:<base64 ciphertext>"
```

加载时先读取 `.config.key`，再依次尝试认证并解析主文件和 `.bak`。解密成功后，每个 provider 条目还必须包含非空 `accessToken` 和字符串 `expiresAt`；任意非法条目都会让整份文件被视为损坏，而不是悄悄丢掉该 provider。

保存时不能简单执行“读旧值→覆盖整个对象”，否则两个 provider 同时刷新 Token 时会互相丢失。Store 会保留调用方最近一次 `load()` 得到的基线快照，把本次对象转换为“新增/修改/删除”差异，然后在文件锁内重新读取磁盘最新值并应用该差异。这样既保留其他进程刚写入的 provider，也能正确删除已过期的 provider。

**安全特性：**
- 文件存储在用户目录下，并使用 AES-256-GCM 认证加密
- `load()` 时做严格的逐 provider 类型校验，非法结构按损坏处理，不会静默丢弃条目
- 主文件损坏或认证失败时读取 `.bak`；主备均无效则失败关闭
- 并发保存通过排他锁、锁内重读与基于已加载快照的增删改合并，避免覆盖其他 provider 的更新
- ENOENT（文件不存在）静默处理为首次授权

## 23. 授权服务器实现（flavor-pkce auth-server）

**通俗理解：** 这一章对应"酒店前台"。它的工作很明确：注册住户（用户管理）、验证住户身份（登录）、确认住户要开哪些房间（授权确认页面）、签发房卡（JWT Token）。下面是它的内部运作方式。

### 23.1 技术栈与职责

授权服务器（`auth-server/auth_server/main.py`）是一个 FastAPI 应用，负责**身份认证 + PKCE 授权 + JWT 签发**。

| 职责 | 说明 |
|------|------|
| 用户注册/登录 | `POST /register`、`POST /login`，使用 bcrypt 哈希密码 |
| PKCE 授权端点 | `GET /authorize`，校验 client + redirect_uri + PKCE 参数 |
| 授权确认页面 | 服务端渲染 consent HTML，展示 client 名、scope、用户名 |
| Token 端点 | `POST /token`，验证 code_verifier + 签发 JWT |
| JWT 生命周期 | RS256 非对称签名、3 天有效期、jti 用于撤销支持 |

### 23.2 数据库 Schema

使用 SQLite 存储，初始化代码在 `auth_server/database.py`：

```sql
-- 注册的 OAuth 客户端（支持 redirect_uri 前缀匹配）
CREATE TABLE clients (
    id            TEXT PRIMARY KEY,      -- 如 "flavor-code-cli"
    name          TEXT NOT NULL,         -- 如 "flavor-code CLI"
    redirect_uris TEXT NOT NULL,         -- JSON array: ["http://127.0.0.1:"]
    created_at    TEXT NOT NULL
);

-- 用户（密码 bcrypt 哈希）
CREATE TABLE users (
    id            TEXT PRIMARY KEY,
    username      TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    created_at    TEXT NOT NULL
);

-- 授权码（一次性、PKCE 绑定、10分钟过期）
CREATE TABLE authorization_codes (
    code             TEXT PRIMARY KEY,
    client_id        TEXT NOT NULL REFERENCES clients(id),
    redirect_uri     TEXT NOT NULL,
    code_challenge   TEXT NOT NULL,      -- SHA256(code_verifier)
    user_id          TEXT NOT NULL REFERENCES users(id),
    scope            TEXT,
    expires_at       TEXT NOT NULL,      -- 10 分钟有效
    used             INTEGER NOT NULL DEFAULT 0,  -- 防重用
    created_at       TEXT NOT NULL
);

-- 已签发 Token（支持撤销查询）
CREATE TABLE tokens (
    id              TEXT PRIMARY KEY,
    jti             TEXT NOT NULL UNIQUE, -- JWT ID
    client_id       TEXT NOT NULL REFERENCES clients(id),
    user_id         TEXT NOT NULL REFERENCES users(id),
    scope           TEXT,
    expires_at      TEXT NOT NULL,
    revoked         INTEGER NOT NULL DEFAULT 0,
    created_at      TEXT NOT NULL
);
```

种子数据自动创建 `flavor-code-cli` 客户端（redirect_uris 匹配 `http://127.0.0.1:` 前缀）和测试用户 `testuser`。

### 23.3 JWT 签发

`auth_server/jwt_utils.py` 使用 RS256（RSA 2048 位）非对称算法：

```python
def create_jwt(sub: str, client_id: str, scope: str = "") -> str:
    payload = {
        "sub": sub,           # 用户 ID
        "client_id": client_id,
        "scope": scope,       # "models:read models:use"
        "iat": timestamp,     # 签发时间
        "exp": timestamp+3天, # 过期时间
        "jti": uuid4(),       # 唯一 ID（撤销时按此匹配）
    }
    return jwt.encode(payload, private_key, algorithm="RS256")
```

**密钥生成机制：**
- 首次运行时自动生成 `keys/private.pem` 和 `keys/public.pem`
- Docker 部署时通过 volume `keys-volume` 在容器间共享

### 23.4 授权端点流程

**通俗理解：** 下面这张图展示了一次完整的"登录→授权→拿房卡"过程。核心要点在于：

1. 用户先登录、再授权——两步分开，防止恶意应用冒充用户
2. 授权码只能使用一次（`used=1`），用完即废——即使被截获也无法重放
3. PKCE 验证在最后一步：服务器算一遍 `SHA256(code_verifier)`，对比之前存下的 `code_challenge`，一致才给 Token——这个过程用的是常量时间比较，防止攻击者通过"多久返回结果"来猜测正确答案

```mermaid
sequenceDiagram
    participant CLI as flavor-code
    participant A as /authorize
    participant DB as SQLite
    participant S as Session
    participant C as /consent
    participant T as /token

    CLI->>A: GET /authorize (response_type=code, client_id, code_challenge, ...)
    A->>DB: 查询 client_id
    DB-->>A: client 有效
    A->>A: 校验 redirect_uri 前缀匹配
    A->>S: get_current_user() 检查 session cookie

    alt 未登录
        A-->>CLI: 302 重定向到 /login
        Note over CLI: 用户在浏览器中登录
        S-->>A: login 成功，设置 session cookie，重定向回 /authorize
    end

    A-->>CLI: 200 HTML consent 页面，展示 client_name, scope, username
    CLI->>C: POST /consent (用户点击 Approve)

    C->>C: 从 session 读取 pending_auth
    C->>DB: INSERT authorization_codes (code, code_challenge, user_id, ...)
    C-->>CLI: 302 重定向到 callback (code + state)

    CLI->>T: POST /token (grant_type, code, code_verifier, redirect_uri, client_id)
    T->>DB: SELECT authorization_codes WHERE code = ?
    T->>T: 校验: 未使用, 未过期, client_id 一致, redirect_uri 一致
    T->>T: PKCE: SHA256(code_verifier) == code_challenge (常量时间比较)
    T->>T: 签发 JWT (RS256)
    T->>DB: UPDATE code.used=1, INSERT tokens
    T-->>CLI: access_token + token_type + expires_in
```

### 23.5 安全设计要点

- **密码存储**：bcrypt 哈希，带 salt
- **Session 管理**：httponly cookie + 3 天过期，防止 XSS 窃取
- **redirect_uri 前缀匹配**：`json.loads(client.redirect_uris)` 后逐个前缀匹配，允许 flavor-code 的动态端口
- **防重放**：authorization_code 使用后标记 `used=1`
- **防时序攻击**：`_constant_time_compare()` 逐字节异或比较 code_challenge
- **CORS**：仅允许配置的域名来源

## 24. API 网关 SSE 代理实现（flavor-pkce api-gateway）

**通俗理解：** 这是整个认证链条中"干活最多"的环节，对应"楼层服务台"。当 flavor-code 带着 JWT（房卡）来请求 LLM 时，网关做五件事：验房卡真假 → 确认没过期 → 把房卡收起来 → 从保险柜取出万能卡 → 用万能卡打开房间门。

最关键的一点：LLM 响应是**流式**的——模型是一个字一个字往外蹦的，不是等全文生成完再一次性返回。所以网关不能等"收齐了再转发"，而是"收到一个字就立刻转发一个字"。这就是 SSE（Server-Sent Events）代理的核心挑战——零缓冲、逐块转发，用户感觉不到中间还有一个转发层。

### 24.1 技术栈与职责

API 网关（`api-gateway/gateway/main.py`）是一个 FastAPI 应用，作为 LLM API 的**透明反向代理**，核心职责是 **JWT 校验 → Token 交换 → 流量转发**。

```mermaid
flowchart LR
    subgraph "API Gateway 五步处理"
        S1["1. 提取 JWT<br/>x-api-key 或 Authorization: Bearer"]
        S2["2. 校验 JWT<br/>RS256 公钥验签 + 过期检查"]
        S3["3. 剥离用户鉴权头<br/>移除 x-api-key, authorization"]
        S4["4. 注入真实 API Key<br/>x-api-key:真实Key（Anthropic SDK）"]
        S5["5. 转发到上游<br/>支持 SSE + 普通请求"]
    end

    S1 --> S2 --> S3 --> S4 --> S5
```

### 24.2 JWT 验证中间件

```python
def verify_jwt(token: str) -> dict | None:
    public_key = load_public_key()  # 从 keys/public.pem 加载，首次缓存
    return jwt.decode(token, public_key, algorithms=["RS256"])
```

- 只做**签名验证 + 过期检查**，不查数据库（零延迟）
- 公钥通过 Docker volume 以只读方式挂载（`:ro`），防篡改

### 24.3 SSE 流式代理

**通俗理解：** 这是网关最"秀技术"的地方。LLM 不是等全文写好再发——它写一个字就发一个字（就像打字机），这叫流式响应。网关必须做到"接到一个字就立刻转发一个字"，不能傻等收集完一整句话再发——那样用户看到的 AI 回复就会像卡顿的视频一样一顿一顿的。

API 网关最关键的挑战是 **Server-Sent Events (SSE) 流式转发**——LLM 响应往往以 `text/event-stream` 格式逐个 token 推送，代理必须逐块转发而不能缓冲整个响应。

```python
@app.api_route("/{path:path}", methods=["GET","POST","PUT","DELETE","PATCH","OPTIONS"])
async def proxy(request: Request, path: str):
    """透明代理。区分 SSE 和普通响应。"""

    # ... JWT 验证 + Header 替换 ...

    async with httpx.AsyncClient(timeout=300.0) as client:
        async with client.stream(method=..., url=upstream_url, ...) as upstream_resp:
            content_type = upstream_resp.headers.get("content-type", "")

            if "text/event-stream" in content_type:
                # SSE: 使用 StreamingResponse 逐块转发
                return StreamingResponse(
                    sse_generator(),           # async generator
                    status_code=status_code,
                    headers=_clean_response_headers(headers, streaming=True),
                )

            # 非 SSE: 完整读取后返回
            content = await upstream_resp.aread()
            return Response(content=content, status_code=...)
```

**SSE 代理核心机制：**

```mermaid
sequenceDiagram
    participant FC as flavor-code<br/>OpenAI Adapter
    participant GW as API Gateway
    participant UP as Upstream OpenAI

    FC->>GW: POST /v1/responses<br/>Authorization: Bearer eyJ...<br/>Accept: text/event-stream
    GW->>GW: 校验 JWT ✓
    GW->>GW: 替换 header → x-api-key:真实Key
    GW->>UP: POST /v1/responses<br/>x-api-key:真实Key<br/>Accept: text/event-stream

    UP-->>GW: HTTP 200<br/>Content-Type: text/event-stream

    loop SSE 流式响应
        UP-->>GW: data: {"type":"response.output_text.delta","delta":"你好"}\n\n
        GW-->>FC: data: {"type":"response.output_text.delta","delta":"你好"}\n\n
        UP-->>GW: data: {"type":"response.output_text.delta","delta":"，世界"}\n\n
        GW-->>FC: data: {"type":"response.output_text.delta","delta":"，世界"}\n\n
        UP-->>GW: data: [DONE]\n\n
        GW-->>FC: data: [DONE]\n\n
    end
```

**设计细节：**

1. **零缓冲转发**：`async for chunk in upstream.aiter_bytes()` 逐块 yield，不做任何拼接
2. **独立生命周期**：SSE generator 内部重新建立 httpx 连接，确保 Starlette 异步读取期间连接保持活跃
3. **长时间超时**：`timeout=300.0`，LLM 完整响应可能需要分钟级时间
4. **Header 清洗**：移除 `transfer-encoding`、`content-encoding`、`content-length`（chunked 传输不需要）
5. **Content-Type 检测**：通过 `text/event-stream` 子串匹配决定路由策略

### 24.4 双 SDK 兼容

API 网关同时支持两种上游鉴权方式：

| SDK | 用户 Header | 上游 Header |
|-----|------------|------------|
| OpenAI | `Authorization: Bearer <JWT>` | `Authorization: Bearer <REAL_KEY>` |
| Anthropic | `x-api-key: <JWT>` | `x-api-key: <REAL_KEY>` |

通过 `UPSTREAM_API_KEY` 环境变量统一配置真实 API Key。

## 28. 对抗性审查方案（/goal 命令）

### 28.0 通俗理解：什么是对抗性审查？

我们让一个 AI"干活"，谁来验证它真的干好了？传统的做法是让干活的那个 AI 自己检查自己——这就像让学生自己批改自己的卷子，难免"放水"。**对抗性审查**的核心思想是：**干活的和审查的是不同的角色**——

1. **Planner（规划者）**：把模糊需求翻译成"可验证的契约"——不是指导你怎么写代码，而是定义"做完之后能观察到什么"
2. **Worker（执行者）**：在契约约束下干活，提交证据
3. **Skeptic Panel（质疑团）**：3 个独立的"刺头"AI 并行审查工作成果，默认立场是"我不信，证明给我看"

如果多数质疑者认为目标已达成，Goal 才算完成。任一阶段不及格，Worker 会被送回重做——附带质疑团提供的具体缺陷清单。这就好比：程序员写代码 → 三个资深审查者交叉检查 → 有 bug 就打回重做 → 重复直到通过。连续多轮无进展则自动熔断。

使用方式：

```text
/goal 修复项目里所有的 TypeScript 类型错误，并确保 npm test 全部通过
```

### 28.1 三阶段流水线

```mermaid
flowchart TD
    START["/goal objective"] --> PLAN["阶段 1: Planner<br/>制定可验证的验收标准"]
    PLAN -->|成功| PLANFILE["写入 .flavor/goal-plan.md"]
    PLAN -->|失败| FAIL_PLAN["goal-failed"]
    PLANFILE --> WORKER["阶段 2: Worker<br/>执行代码变更，收集证据"]
    WORKER --> EVIDENCE["收集证据包：<br/>变更文件 + 最终回复 + 残留 gaps"]
    EVIDENCE --> SKEPTIC["阶段 3: Skeptic Panel<br/>3 个独立 AI 对抗性审查"]
    SKEPTIC --> VOTE["多数投票聚合"]
    VOTE -->|achieved| SUCCESS["goal-complete ✓"]
    VOTE -->|not_achieved| FEEDBACK["反馈具体缺陷清单<br/>送回 Worker 重做"]
    FEEDBACK -->|新 round| WORKER
    VOTE -->|blocked| BLOCKED["goal-blocked<br/>全部质疑者认为不可修复"]
    FEEDBACK -->|stall streak ≥ 2 且指纹不变| STALLED["goal-stalled ✗<br/>自动熔断"]
```

**关键参数（硬编码在 `src/production.ts` 中）：**

| 参数 | 值 | 说明 |
|------|------|------|
| `skepticCount` | 3 | 质疑团人数（1–5） |
| `maxRounds` | 5 | 最大执行-验证轮次 |
| `maxStallStreak` | 2 | 连续停滞轮数上限（相同指纹触发熔断） |
| Planner 模型 | 主 Agent 模型 | 规划阶段共享主力模型 |
| Classifier 模型 | 主 Agent 模型 | 审查阶段共享主力模型 |

### 28.2 阶段一：Goal Planner——制定可验证的契约

**职责：** 将自然语言目标翻译成结构化验收标准。

Planner 被注入如下角色提示（`src/goal/planner.ts`）：

- **只描述 OUTCOME，不描述实现**：定义可观测的结果，不指定模块布局、类名、函数签名
- **验收标准分类**：
  - `gating`（门槛）：必须通过，否则 Goal 被证伪
  - `evidence`（证据）：尽力而为，仅在所有 gating 通过后才检查
- **3–8 条标准**，少而精——更多标准不代表更好的验证
- **验证计划**：明确写出审计者应该看什么、在哪里、期望什么输出

Planner 返回的 JSON 结构（`Plan`）：

```typescript
{
  kind: "code-change" | "analysis" | "research",
  criteria: [
    { id: 1, description: "npm run typecheck exits 0", type: "gating" },
    { id: 2, description: "npm test 全部通过", type: "gating" },
    { id: 3, description: "无 console.log 残留", type: "evidence" }
  ],
  verificationPlan: "运行 npm run typecheck，确认退出码为 0；运行 npm test...",
  approach: "1-2 句实现建议（非契约性，仅给 Worker 参考）",
  checklist: ["bite-sized task 1", "bite-sized task 2"],
  nonGoals: ["不修改测试文件", "不添加新依赖"],
  assumedScope: ["项目使用 TypeScript strict 模式"]
}
```

该计划被写入 `.flavor/goal-plan.md`，作为 Worker 和 Skeptic 的工作契约。

### 28.3 阶段二：Worker 执行——受约执行与证据收集

**职责：** 在契约约束下完成代码变更。

Worker 是一个全新的 `LocalHarness` / `ContextManager` 实例，每次 cycle 独立启动。它收到的 prompt 结构（`buildWorkerPrompt`）包含：

1. **Goal（当前 round）**：原始目标
2. **Acceptance Criteria（契约）**：按编号列出，标注 `[gating]/[evidence]`
3. **Implementation Guidance（非契约性）**：Planner 的建议
4. **Task Checklist**：可选的 bite-sized 任务清单
5. **Prior Verification Gaps（上一轮审查发现的缺陷）**：精确列出需要修复的问题，附 `blocking` 类型

Worker 的关键约束：
- 必须实施并**自行验证**
- 不能仅声称完成——需要证据
- 完成声明后，由独立验证团审计（Worker 不能自己批准成功）
- 如果上一轮有残留 gaps，必须**优先修复**而不是引入新变更

Worker 执行完毕后，系统从当前 workspace 收集**证据包**（`EvidencePacket`）：

| 证据项 | 来源 |
|------|------|
| 变更文件列表 | 扫描 workspace（排除 `.git`、`node_modules`、`.flavor`、隐藏文件） |
| 最终响应 | Worker 最后 4000 字符的文本回复 |
| Plan 文件 | `.flavor/goal-plan.md` 内容 |
| 历史 gaps | 上一轮审查发现的缺陷描述 |

### 28.4 阶段三：Skeptic Panel——对抗性验证

**职责：** 3 个独立的"质疑者"并行审查证据，判定目标是否达成。

每个 Skeptic 的角色提示（`buildSkepticPrompt`，`src/goal/classifier.ts`）的核心指令：

```
You are an adversarial verifier (skeptic #N).
Your job is to AUDIT the work and determine whether the objective has been met.

Default to refuted: true if uncertain.
A false-positive is worse than one more iteration.

Rules:
1. The OBJECTIVE is the immutable contract. Criteria are a derived checklist.
2. If claims are not reflected in CHANGED_FILES → refute (fabrication).
3. TODO/FIXME/unimplemented/skipped tests → refute.
4. Missing tests alone are NOT grounds for refutation unless explicitly required.
5. NEVER invent requirements beyond the contract.
6. In repeat rounds, focus on PRIOR_GAPS — new issues only if clearly demonstrable.
7. Audit CURRENT workspace state — do not re-execute code unless a cheap spot-check.
```

每个 Skeptic 独立审查证据包——它们**看不到对方的输出**，保证审查独立性。Skeptic 返回统一的 Verdict 结构：

```typescript
{
  refuted: boolean,
  gaps: [
    {
      criterion: "AC-2",
      description: "npm test 仍然有 3 个失败用例",
      blocking: "model_fixable" | "contradiction" | "unverifiable"
    }
  ]
}
```

**blocking 三种类型：**

| blocking 值 | 含义 | 示例 |
|-------------|------|------|
| `model_fixable` | Worker 新一轮迭代可修复 | 测试未通过、文件未保存 |
| `contradiction` | 目标与现实矛盾 | 要求删除的文件根本不存在 |
| `unverifiable` | 环境限制无法验证 | 缺少必要的外部服务 |

### 28.5 多数投票与聚合策略

`aggregate()` 函数（`src/goal/classifier.ts`）实现带故障安全（fail-safe）的多数投票：

```mermaid
flowchart TD
    INPUT["N 个 Verdict"] --> COUNT{"refuted 数量 ≥ N/2 ?"}
    COUNT -->|否| ACHIEVED["type: achieved<br/>goal-complete"]
    COUNT -->|是| ALL_BLOCKING{"所有 refuter 的 gaps<br/>全部是 contradiction 或 unverifiable ?"}
    ALL_BLOCKING -->|是| BLOCKED["type: blocked<br/>所有质疑者认为不可修复"]
    ALL_BLOCKING -->|否| NOT_ACHIEVED["type: not_achieved<br/>收集所有 gaps → 去重 → 计算指纹"]
```

关键设计决策：

- **过半通过即达成**：N=3 时，至少需要 2 个 Skeptic 确认才通过
- **Fail-open 策略**：如果某个 Skeptic 返回无法解析的响应，自动按 `refuted: false`（不通过）处理——宁可多跑一轮，也不能因为解析错误放过问题
- **Gap 去重**：同一 criterion + description 的 gap 只保留一条

#### 28.5.1 指纹计算详解

聚合出 `not_achieved` 结果时，系统会为当前 gaps 集合计算一个**内容指纹**（`computeFingerprint`，`src/goal/classifier.ts`），供后续停滞检测使用。

**实现代码：**

```typescript
export function computeFingerprint(gaps: Gap[]): string {
  const content = gaps
    .map((g) => `${g.criterion}:${g.description}`)
    .sort()
    .join("\n");
  return createHash("sha256").update(content).digest("hex").slice(0, 16);
}
```

**逐步拆解：**

```
输入 gaps = [
  { criterion: "AC-2", description: "npm test 仍有 3 个失败", blocking: "model_fixable" },
  { criterion: "AC-1", description: "src/utils.ts 存在类型错误", blocking: "model_fixable" },
]

Step 1 — map 为正则形式
  ["AC-2:npm test 仍有 3 个失败", "AC-1:src/utils.ts 存在类型错误"]

Step 2 — sort() 排序（保证顺序无关）
  ["AC-1:src/utils.ts 存在类型错误", "AC-2:npm test 仍有 3 个失败"]

Step 3 — join("\n") 拼接
  "AC-1:src/utils.ts 存在类型错误\nAC-2:npm test 仍有 3 个失败"

Step 4 — SHA256 哈希
  "a3f1b9c0d2e45678..."  (64 hex chars)

Step 5 — slice(0, 16) 截取前 16 字符
  "a3f1b9c0d2e45678"
```

**设计决策：**

| 决策 | 原因 |
|------|------|
| **排序后再哈希** | 3 个 Skeptic 返回 gaps 的顺序不可控——Skeptic A 可能先报告 AC-2 再 AC-1，Skeptic B 反过来。排序后再 `join` 保证相同的 gap 集合无论以何种顺序到达，都产生相同的指纹 |
| **只取 criterion + description** | `blocking` 类型（`model_fixable` / `contradiction` / `unverifiable`）会随 Worker 修复进度变化——如果 Worker 把一个 `contradiction` 变成 `model_fixable`，gap 本质没变，指纹也不该变。`blocking` 不参与指纹，只影响聚合阶段的 `blocked` 判定 |
| **SHA256 而非短哈希** | 需要抗碰撞的安全哈希——gaps 数量可能达到 8–16 条，简单 CRC 或 32-bit 哈希碰撞概率过高。用 SHA256 后截取前 16 hex 字符（64 bits），碰撞概率 ≈ 1/2^64，在实际 gap 规模下几乎为零 |
| **16 字符而非完整 64 字符** | 指纹仅用于同一 goal 会话内的停滞检测，不是持久化到跨会话比较。64 bits 空间已经远超 5 轮 × 32 条 gap 的可能组合数。截短降低日志和事件中携带的噪音 |

### 28.6 停滞检测与熔断

当 Worker 反复打回重做但毫无进展时，系统需要自动熔断——否则每一轮都在烧模型调用费用却原地踏步。

**核心逻辑**（`src/goal/orchestrator.ts`）：

```typescript
// 每轮 not_achieved 后
if (outcome.fingerprint === priorFingerprint) {
  stallStreak++;
  if (stallStreak >= maxStallStreak) {
    yield { type: "goal-stalled", ... };
    yield { type: "goal-failed", ... };
    return;
  }
} else {
  stallStreak = 1;  // 指纹变化 → 有进展 → 重置计数器
}
priorGaps = outcome.gaps;
priorFingerprint = outcome.fingerprint;
```

**具体场景示例（假设 3 轮执行-验证循环）：**

```
Round 1: Worker 尝试修复 → Skeptic 发现 3 个 gaps
  指纹 = "a3f1b9c0d2e45678"
  priorFingerprint = "" (初始为空)
  不匹配 → stallStreak = 1

Round 2: Worker 被反馈"修复 AC-1 和 AC-2"，但它只修了 AC-1
  Skeptic 仍发现同样 3 个 gaps
  指纹 = "a3f1b9c0d2e45678" (相同!)
  匹配 → stallStreak = 2
  stallStreak ≥ maxStallStreak(2) → goal-stalled → goal-failed ✗
```

**对比：Worker 有进展时：**

```
Round 1: 3 个 gaps → 指纹 "a3f1..."
Round 2: Worker 修复了 AC-1，只剩 AC-2 → 指纹 "b7c2..."
  不匹配 → stallStreak 重置为 1（即使仍有残留问题，也算有进展）
Round 3: 如果 AC-2 又没修好，指纹 "b7c2..." 不变
  匹配 → stallStreak = 2 → 熔断
```

**设计要点：**

- **"部分修复"算进展**：只要指纹变化（gap 增删或描述改变），streak 就重置——不要求零缺陷，只要求不原地踏步
- **gap 增加也算进展**：如果 Worker 引入新的缺陷（指纹变了但可能更糟），计数器也会重置——这不是 bug。停滞检测只判断"动没动"，不判断"方向对不对"。如果 Worker 越改越烂，Skeptic 会在后续轮次中抓到更多 gaps，最终会在某一轮因指纹不变而熔断
- **`goal-stalled` 是温和信号**：它是一个独立的运行时事件，UI 可以展示为"检测到停滞"，随后立即转为 `goal-failed` 终止流水线。两层事件让调用方有机会插入日志或通知
