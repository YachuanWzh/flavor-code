# flavor-code 技术方案报告

> 版本：0.1.0 | 语言：TypeScript | 运行时：Node.js ≥20 | 包管理器：npm

---

## 1. 项目概述

**flavor-code** 是一个运行在终端里的 AI 编程助手。它不像传统 IDE 插件那样需要图形界面——你只要在命令行里和它对话，它就能阅读代码、搜索文件、修改项目、执行命令、拆分复杂任务并调度多个子 Agent 并行工作，就像一个坐在你旁边的资深程序员。

从架构上讲，它是一个实现了 "ReAct 循环"（Reasoning + Acting）的本地 Agent 框架：接入大语言模型（OpenAI GPT、Anthropic Claude 或兼容服务），在工作区权限范围内安全操作文件，通过分层上下文压缩突破单次模型窗口限制，并通过任务 DAG 实现结构化多步协作。

本报告面向开发者、架构师以及希望深入理解 Agent 系统的技术人员，从目标定位、架构设计、核心子系统到安全策略逐一展开。

## 2. 目标与非目标

### 2.1 0.1.0 版本目标

- 提供 `npm install -g flavor-code` 一键安装的终端编程 Agent
- 统一接入多种模型 provider（OpenAI、Anthropic、OpenAI 兼容）
- 以可取消的流式循环调用本地工具（文件读写、搜索、Shell 等）
- 在权限与 Hook 边界内安全运行
- 支持受限于 workspace 权限的子 Agent 进行 DAG 任务并行
- 通过分层上下文压缩跨越单次模型窗口
- 按需加载 Skill（技能包）与可信插件
- 通过 `/loop <goal>` 运行具备宿主验证、预算门槛、隔离与持久化证据的 loop-engineering 外循环
- 在 Windows / macOS 上验证构建和安装

### 2.2 非目标（明确排除）

- 后台 Session Memory（持久化记忆系统）
- Anthropic cache editing
- 压缩后的文件/Skill 附件恢复
- Partial compact UI（渐进式压缩界面）
- IDE 原生协议/扩展
- 云端会话同步 / 多设备协作
- 操作系统级沙箱
- 精确 tokenizer（使用字符估算兜底）
- 授权服务器端实现（委托给 flavor-pkce 项目）

## 5. Agent 核心循环

### 5.1 迭代控制

`AgentLoop.run()` 是一个 `async *` 生成器，输出 `AgentEvent` 事件流，由消费方（UI 或子 Agent 调用者）通过 `for await...of` 消费。

```mermaid
stateDiagram-v2
    [*] --> Running: submit(prompt)
    Running --> Warning: 迭代 ≥ maxIterations × 0.8
    Warning --> Running: 继续
    Warning --> Extended: 达到硬限制 & hasActiveProgress()
    Extended --> Running: 追加 extendBy 轮（最多 3 次）
    Extended --> Error: 扩展耗尽
    Running --> Done: 无 tool call + 纯文本
    Running --> ToolExec: 有 tool call
    ToolExec --> Running: 下一轮迭代
    Running --> Overflow: context_overflow（无产出）
    Overflow --> Running: 反应式压缩 + 重试
    Running --> Error: 取消 / 异常
```

**软限制**：迭代达到 80% 时发出 `warning` 事件，告知用户剩余轮数有限。

**硬限制 + 自动扩展**：到达上限时检查 `hasActiveProgress()`——如果存在 `in_progress` 的主任务或 `running` 的子 Agent，自动追加扩展轮数（默认 20 轮，最多 3 次扩展）。子 Agent 不可扩展。

### 5.2 单轮循环详解

```
prepareForModelCall()
  ├── 自动微压缩（快速清理旧工具输出）
  ├── needsCompaction() 判断
  │     └── 完整压缩（模型摘要）
  └── 返回 messages + system
       ↓
BeforeModelCall hook（可 deny）
       ↓
adapter.stream(model, messages, tools)
  ├── text → yield text 事件
  ├── tool-call → 暂存
  ├── usage → 回写 ContextManager
  └── context_overflow → 反应式压缩 + 重试（最多 1 次）
       ↓
AfterModelCall hook
       ↓
有 tool calls?
  ├── 是 → 逐个执行
  │     ├── PreToolUse hook
  │     ├── PermissionEngine.decide()
  │     ├── tool.execute()
  │     ├── PostToolUse hook
  │     └── 失败时 PostToolUseFailure → AuditLogger
  │     └── appendMany(assistant + tool results)
  └── 否 → append(assistant message) → done
```

关键设计细节：

- **整轮暂存**：工具调用整轮暂存后一次性 `appendMany()` 提交 assistant + tool 消息，保证 provider 对话结构有效
- **反应式压缩**：正常请求在尚未产生文本或工具调用时若返回 `context_overflow`，循环在同一 iteration 内强制压缩并重建请求，最多重试一次。若已有可见输出则不重试，避免拼接
- **取消安全**：通过 `AbortSignal` 支持取消，取消后立即合成错误结果并清理

### 5.3 Loop Engineering 外循环

#### 调研结论与边界

截至 2026-07-15，外部资料对 loop engineering 的共同定义不是“把同一个 prompt 无限重试”，而是设计 Agent 外部的控制系统：明确目标、状态、权限、独立验证、反馈路径、停止规则和记忆。论文 *Stop Hand-Holding Your Coding Agent* 将可复用 loop specification 概括为 trigger、goal、verification、stopping rule 和 memory；[Loop Engineering](https://loopengineering.sh/) 也强调它是围绕 Agent 的控制系统。OpenAI 对 [Codex agent loop](https://openai.com/index/unrolling-the-codex-agent-loop/) 的说明则对应本项目已有的内部“模型—工具—观察”循环；[Harness engineering](https://openai.com/index/harness-engineering/) 展示了在其外层通过审查与反馈继续迭代的实践。

因此本项目明确区分两层：

- `AgentLoop` 是一次 worker 内部的 ReAct 循环，模型无工具调用时即可结束。
- `LoopOrchestrator` 是宿主持有的外循环，worker 的“完成”只表示请求验证；只有宿主验证器通过才产生 `succeeded`。

这种分层也用于控制无限循环风险。[When Agents Do Not Stop](https://arxiv.org/abs/2607.01641) 指出，无有效边界的 Agent 循环会放大成本、上下文增长和重复副作用，因此实现必须同时具备预算门槛、可取消性、无进展熔断和具名终态。

#### 命令与运行流程

用户只需调用：

```text
/loop <goal>
```

`loop` 是保留的内置命令，插件或普通 Skill 不能覆盖。方法论由内置 Loop Skill 注入每个新 cycle；调度、验证、预算、持久化和终态判断属于核心运行时，不能由模型自行批准。

```mermaid
flowchart TD
    START["/loop goal"] --> RESOLVE["推断目标意图、隔离与验证命令"]
    RESOLVE -->|无预置验证器| DISCOVERY["一次 verifier-discovery cycle"]
    DISCOVERY -->|仍无法可靠推断| HUMAN["needs_human"]
    DISCOVERY -->|建立项目原生检查| VERIFY
    RESOLVE -->|已推断验证器| WORKER["创建全新 LocalHarness / ContextManager"]
    WORKER --> VERIFY["宿主执行确定性验证器"]
    VERIFY -->|通过| SUCCESS["succeeded"]
    VERIFY -->|失败| PROGRESS{"连续 3 次相同失败且指纹不变?"}
    PROGRESS -->|是| STUCK["no_progress"]
    PROGRESS -->|否| BUDGET{"达到 cycle 或 token checkpoint?"}
    BUDGET -->|否| WORKER
    BUDGET -->|是| CONFIRM["QuestionBridge 请求用户确认"]
    CONFIRM -->|继续| EXTEND["仅增加一个原始 tranche"]
    EXTEND --> WORKER
    CONFIRM -->|拒绝| EXHAUSTED["budget_exhausted"]
    CONFIRM -->|非交互/不可询问| HUMAN
```

#### 验证与隔离推断

验证器按稳定顺序读取 `package.json` 中的 `test`、`typecheck`、`lint`、`build`、`smoke:install`，也识别 `FLAVOR.md` 中明确列出的受信任 npm 命令。命令由宿主的 Shell 工具执行，退出码、截断后的 stdout/stderr 和摘要写入 cycle evidence。若启动时没有确定性命令，调度器不会在 `0 token` 处退出，而是先运行一次 verifier-discovery cycle：worker 检查项目并建立有意义的项目原生检查，宿主随后重新推断并执行。显然无条件成功的脚本（如 `exit 0`、`echo ok`）不会被信任；discovery 后仍没有可靠命令才进入 `needs_human`，不会用模型自述替代证据。

`isolation: "auto"` 的规则是：只读目标使用当前 workspace；代码修改目标及语义不明确的目标要求干净 Git 仓库，并创建 `.worktrees/loop-<id>` 与 `loop/<id>` 分支。`.worktrees/` 只写入仓库本地 `.git/info/exclude`，不修改共享 `.gitignore`。隔离失败时进入 `needs_human`，不会静默降级到主工作区。

#### 分段预算确认

`loop.maxCycles` 与 `loop.maxTokens` 是授权步长，不是整个 loop 的一次性硬上限。初始 checkpoint 等于配置值；达到门槛后只在当前模型流和宿主验证完成的安全边界询问用户，确认后按原始步长增加：

```text
maxTokens = 5,000,000
5,000,000 → 确认 → 10,000,000 → 确认 → 15,000,000 → ...

maxCycles = 20
20 → 确认 → 40 → 确认 → 60 → ...
```

token 统计包括 worker 的所有模型调用以及上下文压缩模型调用；本地 Shell 验证命令不计 token。用户拒绝产生 `budget_exhausted`；非交互模式无法确认时产生 `needs_human`。预算批准记录维度、旧/新 checkpoint、当时累计用量和时间戳。

#### 状态、持久化与当前限制

每次运行写入 `.flavor/loops/<loop-id>/state.json` 和 `events.jsonl`。snapshot 使用临时文件加原子 rename；journal 追加 cycle、预算和终态事件。具名状态包括 `running`、`succeeded`、`failed`、`blocked`、`cancelled`、`budget_exhausted`、`no_progress`、`needs_human`。

当前版本是前台单 loop：不包含后台调度、跨进程自动恢复、自动 merge/push/deploy、容器沙箱或并发 loop 管理。worktree 与分支会保留供用户检查，不自动合并或删除。`blocked` 已进入持久化协议，首版主要通过权限错误和最终失败证据呈现，尚未对所有工具拒绝原因做精细自动分类。

## 6. 上下文管理与压缩系统

### 6.1 消息布局

每次模型请求时，`ContextManager` 按以下层次拼装消息：

```
┌──────────────────────────────────────┐
│  System Prompt（身份/规则/指令）       │  ← #pinnedMessages() 固定层
├──────────────────────────────────────┤
│  FLAVOR.md（项目指南）                 │  ← 可选
├──────────────────────────────────────┤
│  TaskSnapshot（当前任务状态）          │  ← 可选
├──────────────────────────────────────┤
│  Compact Continuation（压缩摘要续接）  │  ← 注入为 user 消息
├──────────────────────────────────────┤
│  近期完整消息（未压缩部分）            │  ← provider-valid messages
└──────────────────────────────────────┘
```

固定层（system、FLAVOR、TaskSnapshot）不参与压缩持久化，每次由新的 ContextManager 重新生成。

### 6.2 三级压缩策略

```mermaid
flowchart TD
    REQ[每次模型请求前] --> CHK{needsCompaction?}
    CHK -->|否| GO[直接调用模型]
    CHK -->|是| MICRO[一级：微压缩<br/>microcompactMessages]
    MICRO --> CHK2{压力是否解除?}
    CHK2 -->|是| GO
    CHK2 -->|否| AUTO{连续失败 < 3 次?}
    AUTO -->|是| FULL[二级：完整压缩<br/>compactConversation]
    AUTO -->|否| GO_WARN[跳过压缩<br/>熔断保护]
    FULL --> GO

    OVERFLOW[模型返回 context_overflow] -->|无产出| REACT[三级：反应式压缩<br/>重置熔断计数器]
    REACT --> GO_RETRY[重试同一轮]
```

**一级 — 微压缩（Microcompact）**

- 在完整压缩前快速释放空间
- 识别已完成的可压缩工具调用（Read、Shell、Grep、Glob、Edit、Write、ApplyPatch 等 11 种）
- 保留最近 5 个工具结果（`microcompactKeepRecentToolResults`），其余替换为 `[Old tool result content cleared]`
- 失败可回滚，不破坏对话

**二级 — 完整压缩（Model Summary）**

- 调用当前角色模型（空工具集）生成结构化摘要
- 九段式摘要提示词：用户意图与技术决策、文件与代码区、错误与修复、问题解决、待办项、当前工作、下一步、可选分析草稿
- 保留最近 10K–40K token 或 5 条文本消息，不切断 tool call/result
- 自动压缩连续 3 次失败后熔断（避免重复消耗 API）
- 手动 `/compact` 和反应式压缩绕过熔断
- 通过 `snapshot()` / `restore()` 保存和恢复压缩边界

**三级 — 反应式压缩（Reactive）**

- agent loop 在 `context_overflow` 且无可见输出时触发
- 强制完整压缩并重试同一轮
- 重置自动压缩失败计数器

### 6.3 工具输出截断

单次工具消息按 `toolOutputChars`（默认 30,000 字符）进行头尾截断，中间插入截断标记。截断方式为保留字符级头和尾，避免丢失关键信息的同时控制上下文窗口占用。

### 6.4 压力估算模型

- 模型 usage 的 input tokens 回写 ContextManager
- 下一轮以真实 usage + 新增消息估算增量作为压力基线
- 与完整上下文估算取较大值
- 有效窗口 = `windowTokens - reservedOutputTokens`（默认 180K）
- 自动压缩阈值 = 有效窗口 - `autoCompactBufferTokens`（默认 167K）
- 首次调用和刚追加工具结果时使用字符/4 保守估算兜底


**调度机制：**

- 最大并行度：`maxSubagents`（1–16，默认 3）
- 基于 `Promise.race` 事件驱动调度

- 每次尝试最多 2 次：第一次不符合 `SubagentResultSchema` 给修复提示，第二次仍非法则合成 failed

**子 Agent 约束：**

- 权限强制 `workspace` 模式
- 工具集合过滤 `Task`、`TaskPlan`、`TaskUpdate`、`AskUserQuestion`（不可委派）
- 独立 `ContextManager`、`ToolRuntime`、`AgentLoop`
- 通过 `TaskOutput` 工具或最终 assistant 消息的 JSON 解析获取结构化结果

## 8. Provider 适配层

### 8.1 统一流协议

所有模型 provider 通过 `ModelAdapter.stream()` 接口统一接入，输出五种事件：

```typescript
  | { type: "text"; text: string }           // 流式文本块
```

### 9.4 Shell 安全分类

`analyzeCommand()` 对每个命令执行多层分析：

- **包装层检测**：`sh -c`、`cmd /c`、`powershell -Command` 等会递归提取内部命令进行再分析
- **模式匹配**：删除命令（`rm -rf`、`del /f`）、磁盘操作（`mkfs`、`format`、`dd`）、仓库破坏（`git push --force`、`git reset --hard`）、权限修改（`chmod 777`）等
- **例行命令**：测试、构建、版本控制、包管理、格式化等常规开发命令
- **降级规则**：不透明命令（复杂管道、重定向）、多语句串联自动降级

### 9.5 工具参数提示 (Tool Hint)

每个工具定义可选实现 `summarize(input) → string | undefined` 方法，用于生成工具调用的简短参数摘要，不包含 ANSI 转义序列。`ToolRuntime.hint()` 在工具开始前解析 input 并调用 summarize，返回纯文本字符串。

`AgentLoop` 将 hint 附加到 `tool-start` 和 `tool-end` 事件的可选 `hint` 字段中，沿事件管道传递至 UI 层。`transcriptReducer` 将 hint 存储在 `TranscriptBlock.hint` 字段，`StatusLine` 组件以灰暗文字 (`dimColor`) 在括号中渲染：` (hint)`。长 hint 使用 `truncate-end` 截断并附加省略号，不会导致换行。

目前已为所有控制工具和搜索工具实现 summarize：

| 工具 | 摘要示例 |
|------|----------|
| Shell | `npm run build`（命令 + 参数拼接） |
| Glob | `pattern: "**/*.ts" in src` |
| Grep | `pattern: /import/ [typescript] in src` |
| TodoWrite | `3/7 done · Implementing cache layer` |
| AskUserQuestion | `2 questions · 6 options` |
| TaskOutput | 直接复用 `summary` 字段内容 |

hint 生成失败时安全降级——输入 schema 校验失败、summarize 抛异常或返回空字符串均视为无 hint，不影响工具执行流程。

### 9.4 LSP 语义代码分析

传统代码搜索（Grep）只能做**文本匹配**——搜 `handleAuth` 可能返回注释、字符串、甚至不同函数里的同名变量。flavor-code 通过 LSP（Language Server Protocol）实现了**语义级代码理解**。

#### 什么是 LSP

LSP 是微软主导的开放协议。核心理念：一种语言只需写一个 Language Server，所有工具（VS Code、Neovim、flavor-code）都能复用。flavor-code 作为 LSP **客户端**，启动各语言的 Language Server 子进程，通过 JSON-RPC 2.0 over stdio 通信，查询代码的语义信息。

#### 三个 LSP 工具

| 工具 | 作用 | 与 Grep 的对比 |
|------|------|---------------|
| `LspFindRefs` | 找到某个符号的所有引用位置 | Grep 搜到的可能是同名变量、注释、字符串；LSP 只返回同一个符号的精确匹配 |
| `LspHover` | 获取光标处符号的类型定义和文档 | Grep 不支持；LSP 直接返回类型签名和 markdown 文档 |
| `LspDiagnostics` | 读取编译器和 linter 的诊断信息 | Grep 不确定是否有错误；LSP 给出精确的行列号和错误描述 |

#### 架构层次

```
Agent（模型）
  ↓ 调用 LspFindRefs / LspHover / LspDiagnostics
ToolRuntime（统一工具执行）
  ↓ execute()
createLspTools(workspace) → ToolDefinition[]
  ↓ 委托
LspManager（接口：findReferences / hover / diagnostics / dispose）
  ↓ 实现
RealLspManager（会话级管理器）
  ↓ 按需懒启动
LspConnection × N（每种语言一个长连接子进程）
  ↓ JSON-RPC 2.0
Language Server 进程（typescript-language-server / pyright / rust-analyzer / gopls）
```

#### 按需懒启动流程

```mermaid
sequenceDiagram
    participant Agent as Agent Loop
    participant Tools as LSP 工具
    participant Mgr as RealLspManager
    participant Conn as LspConnection

    Agent->>Tools: LspFindRefs("src/foo.ts", 10, 5)
    Tools->>Mgr: findReferences(uri, 10, 5)

    alt 该语言的连接已存在
        Mgr->>Conn: findReferences(uri, 10, 5)
    else 首次访问
        Mgr->>Mgr: 检查项目是否有 tsconfig.json
        Mgr->>Conn: spawn("typescript-language-server", ["--stdio"])
        Conn->>LS: Content-Length: ...\r\n\r\n{initialize}
        LS-->>Conn: {capabilities: {...}}

    end

```

#### JSON-RPC 通信协议

Language Server 是一个独立进程，通过 stdin/stdout 通信。每条消息遵循 LSP 标准帧格式：

```
Content-Length: 156\r\n
Content-Type: application/vscode-jsonrpc; charset=utf-8\r\n
\r\n
{"jsonrpc":"2.0","id":1,"method":"textDocument/references","params":{...}}
```

`LspConnection` 内部维护 `LspBuffer` 做流式帧拆分——TCP 是字节流，消息边界靠 `Content-Length` 头部的数字确定，拆分后按 `id` 匹配请求和响应。支持的请求/通知包括：

| 方法 | 方向 | 说明 |
|------|------|------|
| `initialize` | 请求 | 握手，声明客户端能力（hover、references、diagnostics） |
| `initialized` | 通知 | 握手完成 |
| `textDocument/references` | 请求 | 查找引用 |
| `textDocument/hover` | 请求 | 获取悬停信息 |
| `textDocument/diagnostic` | 请求 | 拉取诊断 |
| `shutdown` + `exit` | 请求+通知 | 优雅关闭 |

#### 语言自动检测

`RealLspManager` 根据**文件后缀**匹配语言，再检查工作区内的**根文件**确认是否需要启动：

| 语言 | 文件后缀 | 根文件（任一存在即启动） | Language Server 命令 |
|------|----------|-------------------------|---------------------|
| TypeScript | `.ts`, `.tsx`, `.mts`, `.cts` | `tsconfig.json` | `typescript-language-server --stdio` |
| Python | `.py`, `.pyi`, `.pyx` | `pyproject.toml`, `pyrightconfig.json` | `pyright-langserver --stdio` |
| Rust | `.rs` | `Cargo.toml` | `rust-analyzer` |
| Go | `.go` | `go.mod`, `go.work` | `gopls` |

连接按语言缓存（`Map<language, LspConnection>`），同一种语言的所有文件共享一个长连接，避免重复启动进程。

#### 权限与安全

LSP 三个工具均归类为 `READ_TOOLS`（只读），与 `Read`、`Grep` 同级。工作区内自动放行，无额外审批开销。Language Server 子进程通过 `spawn` 启动，`windowsHide: true`，dispose 时发送 `shutdown` + `exit` 后 `kill` 清理。

### 10.2 事件格式与处理

```typescript
// 事件格式
{ version: 1, type: "PreToolUse", payload: { toolName, input, ... } }

// 决定格式
{ decision: "allow" | "deny" | "ask", reason?, updatedInput?, additionalContext? }
```

**处理管道：**

1. 按注册顺序遍历处理器
2. `deny` → 短路返回
3. `ask` → 聚合到汇总决定
4. `updatedInput` → 替换后续处理器的 payload（管道模式）
5. `additionalContext` → 跨处理器累积
6. 超时 / 异常 → 按 `failurePolicy` 策略（error/allow/deny/ask）兜底

**Shell 处理器**：通过子进程 stdin 传递 JSON 事件，从 stdout 解析决定，最大输出 1MB，进程树终止跨平台兼容。

```mermaid
    E-->>H: module.activate(context) 10s 超时

    H->>C: 注册 hooks / commands / tools
    C-->>H: checks + claim（防命名冲突）
    H->>C: 插件正常工作

    H->>C: disposer 反序执行
    H->>E: AbortController abort
    H-->>H: PluginUnload hook
```

### 12.2 信任模型

- 插件运行在**进程内**，不是沙箱
- 安全边界：文件系统授权回调 + stat 快照防 TOCTOU + 贡献点声明-注册一致性 + 路径逃逸检测
- 当前信任模型：**只加载可信插件**，恶意插件可绕过窄 API
- 未来方向：Worker/process sandbox、签名验证、能力 broker

- **工作区校验**：realpath 比较防符号链接逃逸
- **大小限制**：5MB 上限（`DEFAULT_MAX_SESSION_BYTES`）
- **保存时机**：prompt/tool 轮完成、任务转换、压缩、模型/权限变化、SessionEnd、dispose 边界——避免 token 级写盘

## 15. 失败分类与容错

| 层 | 代表性失败 | 处理策略 |
|----|-----------|----------|
| **启动配置** | JSON/Zod 错误、无 provider | 启动错误或可操作诊断；无密钥可进入 UI |
| **Provider** | auth/rate/context/model/network | `normalizeProviderError` 结构化分类 |
| **模型流** | 无 done/error、迭代耗尽 | `incomplete_stream` / `iteration_limit` |
| **工具** | schema、权限、超时、取消 | ToolResult error + PostToolUseFailure → AuditLogger（`.flavor/audit.jsonl`） |
| **Hook** | deny、ask、超时、handler 异常 | 决定合并与 failurePolicy 兜底 |
| **DAG** | 环/未知依赖/非法结果 | 规划拒绝、修复一次、合成 failed |
| **Plugin/Skill** | manifest、冲突、越界、身份变化 | diagnostic、拒绝激活/读取、清理已注册项 |
| **Session** | 超限、损坏、版本/工作区不兼容 | 拒绝；损坏文件隔离 |
| **UI/进程** | Ctrl+C、关闭期异常 | AbortSignal、幂等 close/dispose |

### 16.1 主要资产

- 工作区代码
- 宿主文件系统
- API 凭据

### 16.2 威胁与缓解

| 威胁 | 缓解措施 |
|------|----------|
| Prompt injection（模型指令注入） | 分层消息设计、固定层不参与压缩 |

- 同进程插件与被批准 shell 具有宿主进程权力

- 模型可能提出有害但看似常规的操作

## 20. PKCE 认证架构总览

flavor-code 从 0.1.0 开始内置了 OAuth 2.0 PKCE（Proof Key for Code Exchange）客户端能力，支持与配套的 **flavor-pkce** 项目（授权服务器 + API 网关）协作，实现企业级"无 API Key 暴露"的 LLM 安全访问方案。

### 20.1 架构拓扑

```mermaid
flowchart TD
    FC["flavor-code CLI<br/>（TypeScript 客户端）"]
    AUTH["授权服务器<br/>auth-server (Python/FastAPI)<br/>端口 8091"]
    GW["API 网关<br/>api-gateway (Python/FastAPI)<br/>端口 8092"]
    UP["上游 LLM<br/>（如 OpenAI API）"]

    FC -->|1. GET /authorize?code_challenge=...<br/>浏览器回调| AUTH
    AUTH -->|2. 302 redirect → callback<br/>code + state| FC
    FC -->|3. POST /token<br/>code + code_verifier| AUTH
    AUTH -->|4. JWT access_token| FC
    FC -->|5. 所有 API 请求<br/>Authorization: Bearer JWT| GW
    GW -->|6. 校验 JWT → 替换为<br/>真实 API Key| UP
    UP -->|7. SSE/JSON 流式响应| GW
    GW -->|8. 透明转发| FC
```

### 20.2 为什么需要 PKCE + 网关

在团队或企业环境中，直接让每个开发者持有 LLM API Key 存在以下风险：

- **凭据泄露**：API Key 一旦被日志记录、代码提交或恶意窃取，攻击者可直接消费企业配额
- **审计空白**：不知道谁在什么时间使用了多少 token
- **密钥轮换困难**：API Key 泄漏后需要通知所有人更换

**PKCE + API 网关** 方案将真正的 API Key 封存在网关后端：

1. 用户通过浏览器登录授权服务器（用户名/密码）
2. 授权服务器验证身份后签发 **JWT**（JSON Web Token）
3. flavor-code 使用 JWT 向 API 网关发起 LLM 请求
4. 网关验证 JWT 后**替换 Authorization header** 为真实 API Key，转发到上游 LLM
5. 真实 API Key **永不离开网关**，不在网络上传输，不出现在任何客户端

### 20.3 两项目分工

| 项目 | 技术栈 | 角色 | 核心文件 |
|------|--------|------|----------|
| `flavor-code` | TypeScript / Node.js | PKCE 客户端 + LLM Agent | `src/auth/oauth.ts`、`src/auth/store.ts`、`src/config/schema.ts`、`src/production.ts` |
| `flavor-pkce` | Python / FastAPI | 授权服务器 + API 网关 | `auth-server/auth_server/main.py`、`api-gateway/gateway/main.py` |

## 21. PKCE 客户端实现（flavor-code）

### 21.1 客户端架构

flavor-code 的 PKCE 客户端由三个模块组成，呈单向依赖关系：

```
src/config/schema.ts ────→ 定义 OAuth 配置字段
src/production.ts ───────→ 启动时调用 OAuth 流程获取 API Key
src/auth/oauth.ts ───────→ OAuthCallbackAuthProvider（PKCE 核心）
src/auth/store.ts ───────→ 文件系统 Token 持久化
src/auth/types.ts ───────→ AuthProvider / AuthResult / ApiKeyAuthProvider
```

### 21.2 配置 Schema 扩展

`ProviderConfigSchema`（`src/config/schema.ts`）在原有 `apiKey` 字段基础上新增了 OAuth PKCE 字段：

```typescript
apiType: z.enum(["openai", "anthropic"]).optional(),  // API 协议类型
authorizationUrl: z.string().url().optional(),         // 授权端点
tokenUrl: z.string().url().optional(),                 // Token 端点
clientId: z.string().min(1).optional(),                // 客户端标识
scope: z.string().optional(),                          // 权限范围
```

### 21.3 Provider 启动决策树

`registerConfiguredAdapters()`（`src/production.ts`）按以下优先级确定 API Key：

```mermaid
flowchart TD
    START["Provider 配置"] --> A{有 apiKey?}
    A -->|是| USE_KEY["直接使用 apiKey"]
    A -->|否| B{OAuth 字段完整?<br/>authorizationUrl + tokenUrl + clientId}
    B -->|是| PKCE["执行 PKCE 流程<br/>OAuthCallbackAuthProvider"]
    B -->|否| C{有 OAUTH_* 环境变量?}
    C -->|是| PKCE_DEFAULT["使用内建默认值<br/>执行 PKCE 流程"]
    C -->|否| D{有对应 API Key 环境变量?}
    D -->|是| USE_ENV["使用环境变量"]
    D -->|否| ERR["诊断错误：缺少凭据"]
```

**关键逻辑**：`resolveOAuthConfig()` 函数在没有显式 apiKey 和 OAuth 配置时，自动从环境变量 `OAUTH_AUTHORIZATION_URL`、`OAUTH_TOKEN_URL`、`OAUTH_CLIENT_ID`、`OAUTH_SCOPE` 读取内建默认值，无需用户手动填写。

### 21.4 OAuthCallbackAuthProvider 流程

`OAuthCallbackAuthProvider`（`src/auth/oauth.ts`）实现了完整的 RFC 7636 PKCE 客户端流程：

```mermaid
sequenceDiagram
    participant STORE as Token Store<br/>(auth.json)
    participant PROV as OAuthCallbackAuthProvider
    participant BROWSER as 系统浏览器
    participant SRV as 临时 HTTP Server<br/>(127.0.0.1:随机端口)
    participant AUTH as 授权服务器

    PROV->>STORE: load() 检查缓存 Token
    alt Token 有效（过期时间 - 60s > 当前时间）
        STORE-->>PROV: { accessToken, expiresAt }
        PROV-->>调用方: AuthResult { Authorization: Bearer JWT }
    end

    STORE-->>PROV: 无有效 Token

    PROV->>PROV: 生成 code_verifier (randomBytes(32) → base64url, 43字符)
    PROV->>PROV: 计算 code_challenge (SHA256(code_verifier) → base64url)
    PROV->>PROV: 生成 state (randomBytes(32) → base64url, 防CSRF)
    PROV->>SRV: 启动临时 HTTP Server 监听 127.0.0.1:随机端口/callback

    PROV->>PROV: 构造 authorizationUrl ?response_type=code &client_id=... &redirect_uri=... &code_challenge=... &state=...

    PROV->>BROWSER: openBrowser(authUrl)
    BROWSER->>AUTH: GET /authorize?...（用户登录+授权）
    AUTH->>BROWSER: 302 重定向到 callback
    BROWSER->>SRV: GET /callback?code=...&state=...

    SRV->>SRV: 校验 state 一致（防 CSRF）
    SRV->>SRV: 校验 code 非空
    SRV-->>PROV: 返回 authorization_code

    PROV->>AUTH: POST /token (grant_type, code, code_verifier, redirect_uri, client_id)

    alt PKCE 验证成功
        AUTH-->>PROV: { access_token: eyJ..., token_type: Bearer, expires_in: 259200 }
        PROV->>STORE: save({ accessToken, refreshToken?, expiresAt, scope })
        PROV-->>调用方: AuthResult { Authorization: Bearer eyJ... }
    else 验证失败
        AUTH-->>PROV: { error: invalid_grant }
        PROV-->>调用方: throw Error
    end
```

**安全设计要点：**

1. **code_verifier 熵**：32 字节随机数 → 256 位熵，暴力破解不可行
2. **S256 散列**：SHA256(code_verifier) → code_challenge，即使 challenge 泄露也无法还原 verifier
3. **state 防 CSRF**：每次流程生成新的 32 字节随机 state，回调时必须原样匹配
4. **重定向 URI 锁定**：固定 `http://127.0.0.1:{随机端口}/callback`，仅接受本地回环
5. **5 分钟超时**：防止悬挂的授权流程无限等待（`PKCE_TIMEOUT_MS = 300_000`）
6. **Token 预刷新**：过期前 60 秒自动丢弃缓存，触发新的 PKCE 流程
7. **同时只允许一个流程**：`#pendingFlow` 防并发授权

### 21.5 Token 持久化

`createFileTokenStore()`（`src/auth/store.ts`）将 Token 持久化到 `~/.flavor-code/auth.json`：

```json
{
  "openai": {
    "accessToken": "eyJhbGciOiJSUzI1NiIs...",
    "expiresAt": "2026-07-17T12:00:00.000Z",
    "scope": "models:read models:use"
  }
}
```

**安全特性：**
- 文件存储在用户目录下，权限由操作系统管理
- `load()` 时做严格的类型校验（`isStoredToken`），拒绝非法结构
- 损坏的 JSON 返回空对象，不抛异常阻断启动
- ENOENT（文件不存在）静默处理为首次授权

## 22. 授权服务器实现（flavor-pkce auth-server）

### 22.1 技术栈与职责

授权服务器（`auth-server/auth_server/main.py`）是一个 FastAPI 应用，负责**身份认证 + PKCE 授权 + JWT 签发**。

| 职责 | 说明 |
|------|------|
| 用户注册/登录 | `POST /register`、`POST /login`，使用 bcrypt 哈希密码 |
| PKCE 授权端点 | `GET /authorize`，校验 client + redirect_uri + PKCE 参数 |
| 授权确认页面 | 服务端渲染 consent HTML，展示 client 名、scope、用户名 |
| Token 端点 | `POST /token`，验证 code_verifier + 签发 JWT |
| JWT 生命周期 | RS256 非对称签名、3 天有效期、jti 用于撤销支持 |

### 22.2 数据库 Schema

使用 SQLite 存储，初始化代码在 `auth_server/database.py`：

```sql
-- 注册的 OAuth 客户端（支持 redirect_uri 前缀匹配）
CREATE TABLE clients (
    id            TEXT PRIMARY KEY,      -- 如 "flavor-code-cli"
    name          TEXT NOT NULL,         -- 如 "flavor-code CLI"
    redirect_uris TEXT NOT NULL,         -- JSON array: ["http://127.0.0.1:"]
    created_at    TEXT NOT NULL
);

-- 用户（密码 bcrypt 哈希）
CREATE TABLE users (
    id            TEXT PRIMARY KEY,
    username      TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    created_at    TEXT NOT NULL
);

-- 授权码（一次性、PKCE 绑定、10分钟过期）
CREATE TABLE authorization_codes (
    code             TEXT PRIMARY KEY,
    client_id        TEXT NOT NULL REFERENCES clients(id),
    redirect_uri     TEXT NOT NULL,
    code_challenge   TEXT NOT NULL,      -- SHA256(code_verifier)
    user_id          TEXT NOT NULL REFERENCES users(id),
    scope            TEXT,
    expires_at       TEXT NOT NULL,      -- 10 分钟有效
    used             INTEGER NOT NULL DEFAULT 0,  -- 防重用
    created_at       TEXT NOT NULL
);

-- 已签发 Token（支持撤销查询）
CREATE TABLE tokens (
    id              TEXT PRIMARY KEY,
    jti             TEXT NOT NULL UNIQUE, -- JWT ID
    client_id       TEXT NOT NULL REFERENCES clients(id),
    user_id         TEXT NOT NULL REFERENCES users(id),
    scope           TEXT,
    expires_at      TEXT NOT NULL,
    revoked         INTEGER NOT NULL DEFAULT 0,
    created_at      TEXT NOT NULL
);
```

种子数据自动创建 `flavor-code-cli` 客户端（redirect_uris 匹配 `http://127.0.0.1:` 前缀）和测试用户 `testuser`。

### 22.3 JWT 签发

`auth_server/jwt_utils.py` 使用 RS256（RSA 2048 位）非对称算法：

```python
def create_jwt(sub: str, client_id: str, scope: str = "") -> str:
    payload = {
        "sub": sub,           # 用户 ID
        "client_id": client_id,
        "scope": scope,       # "models:read models:use"
        "iat": timestamp,     # 签发时间
        "exp": timestamp+3天, # 过期时间
        "jti": uuid4(),       # 唯一 ID（撤销时按此匹配）
    }
    return jwt.encode(payload, private_key, algorithm="RS256")
```

**密钥生成机制：**
- 首次运行时自动生成 `keys/private.pem` 和 `keys/public.pem`
- Docker 部署时通过 volume `keys-volume` 在容器间共享

### 22.4 授权端点流程

```mermaid
sequenceDiagram
    participant CLI as flavor-code
    participant A as /authorize
    participant DB as SQLite
    participant S as Session
    participant C as /consent
    participant T as /token

    CLI->>A: GET /authorize (response_type=code, client_id, code_challenge, ...)
    A->>DB: 查询 client_id
    DB-->>A: client 有效
    A->>A: 校验 redirect_uri 前缀匹配
    A->>S: get_current_user() 检查 session cookie

    alt 未登录
        A-->>CLI: 302 重定向到 /login
        Note over CLI: 用户在浏览器中登录
        S-->>A: login 成功，设置 session cookie，重定向回 /authorize
    end

    A-->>CLI: 200 HTML consent 页面，展示 client_name, scope, username
    CLI->>C: POST /consent (用户点击 Approve)

    C->>C: 从 session 读取 pending_auth
    C->>DB: INSERT authorization_codes (code, code_challenge, user_id, ...)
    C-->>CLI: 302 重定向到 callback (code + state)

    CLI->>T: POST /token (grant_type, code, code_verifier, redirect_uri, client_id)
    T->>DB: SELECT authorization_codes WHERE code = ?
    T->>T: 校验: 未使用, 未过期, client_id 一致, redirect_uri 一致
```

### 22.5 安全设计要点

- **密码存储**：bcrypt 哈希，带 salt
- **Session 管理**：httponly cookie + 3 天过期，防止 XSS 窃取
- **redirect_uri 前缀匹配**：`json.loads(client.redirect_uris)` 后逐个前缀匹配，允许 flavor-code 的动态端口
- **防重放**：authorization_code 使用后标记 `used=1`
- **防时序攻击**：`_constant_time_compare()` 逐字节异或比较 code_challenge
- **CORS**：仅允许配置的域名来源

## 23. API 网关 SSE 代理实现（flavor-pkce api-gateway）

### 23.1 技术栈与职责

API 网关（`api-gateway/gateway/main.py`）是一个 FastAPI 应用，作为 LLM API 的**透明反向代理**，核心职责是 **JWT 校验 → Token 交换 → 流量转发**。

```mermaid
flowchart LR
    subgraph "API Gateway 五步处理"
        S1["1. 提取 JWT<br/>x-api-key 或 Authorization: Bearer"]
        S2["2. 校验 JWT<br/>RS256 公钥验签 + 过期检查"]
        S3["3. 剥离用户鉴权头<br/>移除 x-api-key, authorization"]
        S4["4. 注入真实 API Key<br/>x-api-key:真实Key（Anthropic SDK）"]
        S5["5. 转发到上游<br/>支持 SSE + 普通请求"]
    end

    S1 --> S2 --> S3 --> S4 --> S5
```

### 23.2 JWT 验证中间件

```python
def verify_jwt(token: str) -> dict | None:
    public_key = load_public_key()  # 从 keys/public.pem 加载，首次缓存
    return jwt.decode(token, public_key, algorithms=["RS256"])
```

- 只做**签名验证 + 过期检查**，不查数据库（零延迟）
- 公钥通过 Docker volume 以只读方式挂载（`:ro`），防篡改

### 23.3 SSE 流式代理

API 网关最关键的挑战是 **Server-Sent Events (SSE) 流式转发**——LLM 响应往往以 `text/event-stream` 格式逐个 token 推送，代理必须逐块转发而不能缓冲整个响应。

```python
@app.api_route("/{path:path}", methods=["GET","POST","PUT","DELETE","PATCH","OPTIONS"])
async def proxy(request: Request, path: str):
    """透明代理。区分 SSE 和普通响应。"""

    # ... JWT 验证 + Header 替换 ...

    async with httpx.AsyncClient(timeout=300.0) as client:
        async with client.stream(method=..., url=upstream_url, ...) as upstream_resp:
            content_type = upstream_resp.headers.get("content-type", "")

            if "text/event-stream" in content_type:
                # SSE: 使用 StreamingResponse 逐块转发
                return StreamingResponse(
                    sse_generator(),           # async generator
                    status_code=status_code,
                    headers=_clean_response_headers(headers, streaming=True),
                )

            # 非 SSE: 完整读取后返回
            content = await upstream_resp.aread()
            return Response(content=content, status_code=...)
```

**SSE 代理核心机制：**

```mermaid
sequenceDiagram
    participant FC as flavor-code<br/>OpenAI Adapter
    participant GW as API Gateway
    participant UP as Upstream OpenAI

    FC->>GW: POST /v1/responses<br/>Authorization: Bearer eyJ...<br/>Accept: text/event-stream
    GW->>GW: 校验 JWT ✓
    GW->>GW: 替换 header → x-api-key:真实Key
    GW->>UP: POST /v1/responses<br/>x-api-key:真实Key<br/>Accept: text/event-stream

    UP-->>GW: HTTP 200<br/>Content-Type: text/event-stream

    loop SSE 流式响应
        UP-->>GW: data: {"type":"response.output_text.delta","delta":"你好"}\n\n
        GW-->>FC: data: {"type":"response.output_text.delta","delta":"你好"}\n\n
        UP-->>GW: data: {"type":"response.output_text.delta","delta":"，世界"}\n\n
        GW-->>FC: data: {"type":"response.output_text.delta","delta":"，世界"}\n\n
        UP-->>GW: data: [DONE]\n\n
        GW-->>FC: data: [DONE]\n\n
    end
```

**设计细节：**

1. **零缓冲转发**：`async for chunk in upstream.aiter_bytes()` 逐块 yield，不做任何拼接
2. **独立生命周期**：SSE generator 内部重新建立 httpx 连接，确保 Starlette 异步读取期间连接保持活跃
3. **长时间超时**：`timeout=300.0`，LLM 完整响应可能需要分钟级时间
4. **Header 清洗**：移除 `transfer-encoding`、`content-encoding`、`content-length`（chunked 传输不需要）
5. **Content-Type 检测**：通过 `text/event-stream` 子串匹配决定路由策略

### 23.4 双 SDK 兼容

API 网关同时支持两种上游鉴权方式：

| SDK | 用户 Header | 上游 Header |
|-----|------------|------------|
| OpenAI | `Authorization: Bearer <JWT>` | `Authorization: Bearer <REAL_KEY>` |
| Anthropic | `x-api-key: <JWT>` | `x-api-key: <REAL_KEY>` |

通过 `UPSTREAM_API_KEY` 环境变量统一配置真实 API Key。

### 25.2 安全边界

| 安全属性 | 实现方式 |
|----------|----------|
| 真实 Key 不离开网关 | `UPSTREAM_API_KEY` 仅存在于网关环境变量，网络传输只经过网关→上游的 TLS 隧道 |

| 防 code 重放 | authorization_codes.used 字段一次性标记 |
| 防时序攻击 | code_challenge 比较使用常量时间算法 |
| 密钥隔离 | Docker volume 只读挂载公钥到网关 |
