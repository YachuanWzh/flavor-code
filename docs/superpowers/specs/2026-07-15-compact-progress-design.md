# Compact 进度显示设计

## 目标

当主会话执行完整上下文压缩时，在当前终端对话中立即显示一行进度。进度由十个小格组成：未完成格为灰色，每完成 10% 点亮一个蓝色格；同时显示整数百分比。手动 `/compact`、自动压缩和溢出恢复复用同一条进度链路。

## 数据流

`ContextManager` 在完整压缩的关键阶段上报离散百分比：开始 0%、PreCompact 通过 10%、摘要流开始 20%、摘要文本流每批推进 10%（最高 80%）、摘要完成 80%、PostCompact 完成 90%、事务提交完成 100%。摘要总长度事先不可知，因此 20% 到 80% 表示已观测到的流式处理进展，而不是 token 总量预测。

主 agent 的 `ContextManager` 回调把进度转换成 `compact-progress` 会话事件；子 agent 不向主终端发布该事件。`transcriptReducer` 用固定 block id 原位更新同一行，避免每次进度产生新行。终端组件把百分比归一化到 0–100 的 10% 档位，渲染蓝色已完成格和灰色剩余格。

## 边界与失败

- 只有存在可压缩旧消息时才显示进度；无需压缩时仍保留原提示。
- 回调异常不得打断压缩事务。
- 压缩取消或失败时不伪造 100%；现有错误/取消输出继续说明终止原因。
- 完整压缩的上下文事务语义、Hook 顺序和持久化行为保持不变。

## 测试

- ContextManager 测试验证阶段顺序以及最终 100%。
- summarizer 测试验证流式文本只按 10% 档位推进且不超过 80%。
- transcript 测试验证同一进度 block 被原位更新。
- 终端渲染测试验证恰好十格，并在 30% 时出现三个蓝格、七个灰格。
