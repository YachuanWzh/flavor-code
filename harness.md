# flavor-code Harness Engineering 对照分析

> 以"驾驭工程"五维框架为参照，找出 flavor-code 源码中的 Harness 思想体现。
> **分析范围**：`src/` 项目源码，排除 superharness 插件和 `/loop` 命令系统。

---

## 什么是 Harness Engineering？

简单说：**把 AI Agent 当作一匹力大无穷但容易跑偏的马，Harness 就是缰绳、马鞍、嚼子的总和。**

它不优化模型本身，而是优化模型运行的"外部环境"——告诉 Agent 什么能做、什么不能做，做错了怎么纠正，记性怎么维持，以及如何确保安全。

---

## 一、约束机制——告诉 Agent "别乱跑" ⭐ 8/10

### 做得好

**1. 权限引擎：不是简单的"允许/拒绝"**

`src/permissions/engine.ts`

把 Agent 能用的工具分成六类，不同类别有不同规则：

| 类别 | 举例 | 规则思路 |
|------|------|---------|
| control | TodoWrite、AskUserQuestion | 永远放行，不影响系统 |
| read | Read、Glob、Grep | 默认放行 |
| write | Write、Edit、ApplyPatch | 看权限模式决定 |
| destructive | Delete、Move | 几乎总需要人审批 |
| shell | Shell | 深度分析命令内容 |
| network | WebFetch、WebSearch | Subagent 需要主 Agent 批准 |

还支持五种模式之间切换：可以设成"只读模式"让 Agent 只能看不能改，或者"自动模式"让系统自己判断。

**2. Agent 循环中的"熔断器"**

`src/agent/loop.ts`

就像家里电闸的保险丝，Agent 跑太久了会自动停下来：
- 跑到上限的 80% → 发一个预警
- 跑到上限但工作还在进行 → 自动续 20 轮（最多续 3 次），不打断有用工作
- 真跑满了 → 直接终止

**3. 工具重试熔断**

`src/hallucination/retry-monitor.ts`

如果 Agent 用同一个工具反复失败，系统不会让它无限重试。达到阈值 → 生成违规记录；同一参数反复调用 → 触发断路器，直接判定失败。

### 不足

- Shell 命令直接在真机上执行，没有 Docker 那样的隔离沙箱
- Subagent（子 Agent）的迭代次数写死了，不能像主 Agent 那样动态续命

---

## 二、反馈回路——做错了能发现 ⭐ 7/10

### 做得好

**1. "二次审查"：用小模型检查大模型的输出**

`src/hallucination/guard.ts` + `confidence.ts`

每次 Agent 跑完任务，系统会用**一个便宜的模型**再来做三道检查：

- **任务匹配度**（权重 40%）：你说要做的，它做了吗？
- **证据支撑度**（权重 40%）：它的结论有工具调用的结果撑腰吗？
- **过程可靠性**（权重 20%）：中间有没有没解决的报错？

综合得分太低 → 任务被判定不合格。还会列出最多 3 条"没有依据的声明"。

**2. "黑匣子"：记录 Agent 到底干了什么**

`src/hallucination/evidence-ledger.ts`

Agent 过程中调了哪些工具、传了什么参数、返回了什么结果，全记在一个小本本上（最多 24 条事件）。测试、写入、失败的操作优先保留。事后可以用来排查问题，也给上面的二次审查提供证据。

**3. Agent 循环内的故障恢复**

`src/agent/loop.ts`

执行过程中碰到的错误不是简单的"挂了"：
- 调模型失败（网络问题）→ 指数退避重试（等 1 秒、2 秒、4 秒），最多 3 次
- 3 次都失败 → 换备用模型再试 2 次
- Context 溢出 → 自动压缩对话历史再试一次
- 工具调用参数格式不对 → 用备用模型"修复参数"再执行

### 不足

- 二次审查只在任务全跑完时做一次，中间步骤没有实时检查
- 验证主要看 Shell 命令的 exit code 是不是 0（命令本身正常退出就算过），可能漏掉逻辑错误

---

## 三、工作流控制——把大任务拆成小步骤 ⭐ 8/10

### 做得好

**1. 任务图：不是简单列表，是有依赖关系的 DAG**

`src/agent/planner.ts`

任务可以拆成多个节点，每个节点声明"我必须等谁先完成"。系统会做三件事：
- 检查有没有重复 ID
- 检查有没有不存在的依赖
- 检查有没有**死循环依赖**（A 等 B、B 等 A）

**2. 对话状态持久化：挂了能接上**

`src/session/store.ts`

跟 Agent 的对话会存成文件（`.flavor/sessions/`），用的是：
- JSONL 格式，一条消息一行，新消息直接追加
- 写入方式是"先写临时文件→同步到磁盘→改名替换"，保证要么全写完、要么全没写
- 文件坏了 → 自动隔离成 `.corrupt` 文件，不影响其他数据
- 超过上限 → 自动删最旧的

**3. 多层故障恢复**

`src/agent/loop.ts`

上面反馈回路部分已详述：模型失败 → 换模型 → context overflow → 压缩重试 → 参数修复。多道防线保证不会因为一个小问题整个任务崩掉。

### 不足

- 目前只支持"主子 Agent"两层，做不了更复杂的协同（多个 Agent 同时干活再汇总）
- 持久化虽然做得好，但没有"从上次断点继续"的 resume 能力

---

## 四、记忆与管理——记性不好但要会取舍 ⭐ 7/10

### 做得好

**1. "先省着删，不行再总结"的分层压缩**

`src/context/manager.ts`

Agent 的"记忆"（上下文）是会满的。满了怎么办？分三步：

```
第一步: 轻量清理 (microcompact)
  → 删掉重复的工具调用结果，能省多少省多少
  
第二步: 完整压缩 (compact)
  → 用 LLM 把旧消息总结成一段摘要，旧消息丢弃
  
第三步: 熔断保护
  → 连续压缩 3 次都失败？不压了，继续跑，别卡死
```

什么时候触发压缩也不是拍脑袋的，而是根据当前 Token 使用量和可用窗口大小计算"压力值"。

**2. 工具输出自动截断**

`src/context/manager.ts` (`truncateToolOutput`)

每个工具的输出不能无限塞进对话。超出限制 → 保留头部和尾部各一半，中间标注 `[truncated]`。

**3. Context 溢出后自动自救**

`src/agent/loop.ts`

当模型报"上下文太大"并且这一轮还没产出任何内容时，自动触发一次压缩然后重试——不做无用功。

### 不足

- 旧消息摘要后就真的丢了，不能用"展开"的方式找回原始内容
- 跨对话记忆完全丢失：上次聊的内容，下次打开 Agent 完全不记得
- Token 估算用的是"字数 ÷ 4"的粗略算法，不是精确的 tokenizer

## 五、安全管控——防火墙 ⭐ 8/10

### 做得好

**1. Shell 命令被"层层剥开"检查**

`src/permissions/engine.ts`

Agent 可能会写 `sh -c "bash -c 'rm -rf /'"` 这种嵌套命令。权限引擎会递归解析（最多 4 层），剥开所有壳，看到底要执行什么。然后：
- 禁止 `rm -rf /`、`shutdown`、`format`、`dd of=/dev/*` 等危险操作
- 检测管道 `|`、重定向 `>`、链式执行 `;` `&&`——这些需要审批
- Subagent 额外紧箍咒：禁用 destructive 命令 + 必须指定工作目录 + 参数路径不能逃逸

**2. 路径防护：防止 Agent "越狱"**

`src/permissions/engine.ts`

四道防线确保 Agent 操作的文件全在工作区里：

```
1. 词法检查 → 路径里有没有 ".." 想往上跳
2. 物理解析 → 有没有用软链接偷偷跳出去
3. 逐级解析 → 即使文件夹还不存在，也能算出最终会落到哪
4. 边界判定 → 最终确认在不在工作区内
```

**3. 密钥自动洗掉**

`src/session/store.ts`

存盘时会自动检测并删除 `sk-xxx` 格式的 API Key、`authorization`、`secret` 等敏感信息，不会让密钥不小心写进文件。

**4. Subagent 不能"越级"**

子 Agent 的权限比主 Agent 严格得多：
- 不能用 `Task` 工具（防止递归分派新子 Agent）
- 网络访问需要主 Agent 批准
- 所有路径操作限制在工作区
- 权限模式从主 Agent 继承但**不会更宽松**

### 不足

- 网络内容没有安全过滤：允许访问某个 URL 后，不检查返回内容里有没有恶意内容
- 同一类操作被批准后，后续同类操作都放行，缺少细粒度参数检查
- 没有"限速"：Agent 可以一口气狂发 API 请求和工具调用

---

## 总结

| 维度 | 评分 | 最值得说的 | 最主要问题 |
|------|------|----------|----------|
| 约束机制 | 8/10 | 五模式权限 + 双熔断 + 断路器 | 没沙箱 |
| 反馈回路 | 7/10 | 小模型复查 + 审计黑匣子 + 多级重试 | 只有事后检查 |
| 工作流控制 | 8/10 | DAG 任务图 + 状态持久化 + 故障恢复 | 编排简单 |
| 记忆与管理 | 7/10 | 分层压缩 + 熔断保护 | 跨对话全忘 |
| 安全管控 | 8/10 | Shell 递归解析 + 四层路径防护 + 密钥自洗 | 无内容过滤 |

**一句话评价**：flavor-code 的 Harness 设计已经非常成熟——Agent 能做什么、不能做什么、做错了怎么兜底、记性怎么管理、安全怎么保障，每个维度都有专门的代码模块在支撑。最主要的两块短板是"断点续跑"（状态存了但不会从中间恢复）和"过程检查"（只在跑完后才验证，中间步骤缺乏实时监控）。
